## Coding Standards Archive — Moved to Code Comments

Archived from `.claude/CLAUDE.md` Coding Standards on 2026-03-31. Each rule should become an inline code comment at the noted location.

### `spatial_data.py`
- Spatial metadata values may arrive as strings after JSON round-tripping — cast to `float()` before numeric formatting (`:,.0f`)

### `pricing.css`
- **Page-scoped CSS class naming**: Generic class names in page-specific stylesheets (e.g., `.section-label` in `pricing.css`) can collide with identically-named rules in globally-loaded stylesheets (e.g., `report.css`). If a page-specific class name is generic, prefix it with the page name (`.pricing-section-label`) to prevent silent style bleed if stylesheets are ever co-loaded.
- **New visual sub-palette vs. token reuse**: When a page intentionally introduces a neutral/monochrome sub-palette (e.g., `#111111`, `#666666`) that diverges from the site's blue-tinted token system, use tokens for *structural* values (spacing, font weights, border-radius, transitions) but leave palette colors as explicit values rather than mapping them to semantically mismatched tokens. Forcing `#111111` into `--color-text-primary: #0F172A` (navy) or `#666666` into `--color-text-muted: #64748b` (cool blue-gray) produces subtle hue drift that undermines the design intent. Within the sub-palette, consolidate similar grays (e.g., `#555555` and `#8E8E8E` both → `#666666`) to avoid the "narrow band of grays" AI-slop signal.

### `_macros.html` / `_result_sections.html`
- **Inline SVG icons over HTML entities**: Use inline SVGs (e.g., `<svg width="12" height="12" viewBox="0 0 24 24" ...><polyline .../></svg>`) instead of HTML character entities (`&#9654;`, `&#9888;`, `&#9650;`) for icons. SVGs render crisply at all sizes, inherit `currentColor`, and animate smoothly via CSS `transform`. When replacing entities with SVGs inside an existing `.collapse-icon` span, switch the span from `display: inline-block` to `display: inline-flex` and add `.collapse-icon svg { display: block }` to eliminate baseline gaps. When replacing icons across the codebase, audit ALL templates in one pass — `_result_sections.html`, `compare.html`, `compare_health.html`, `_eval_snippet.html` — to avoid inconsistent rendering between pages.

### `index.html` (near `csrfFetch` definition)
- **CSRF auto-retry pattern**: `csrfFetch(url, options)` in `index.html` wraps `fetch()` with CSRF token injection, `resp.ok`/content-type guarding, and single auto-retry on `error_code === 'csrf_expired'`. Use `csrfFetch` for all new POST endpoints instead of raw `fetch` + manual `X-CSRFToken` header. The server-side `@app.errorhandler(400)` adds `"error_code": "csrf_expired"` to JSON responses when the error message contains "csrf" — client matches on that structured code, not on substring search of the human-readable error text.

### `index.html` + `compare_health.html` (Google Maps)
- **Google Maps Autocomplete (split implementation)**: `index.html` uses the **legacy** `google.maps.places.Autocomplete` widget (visible `#address-input`, `.pac-*` dropdown appended to `<body>`). `compare_health.html` uses the **new** `PlaceAutocompleteElement` (shadow DOM, `gmp-select` event, `fetchFields()` required). Both pages use the Google Dynamic Library Import bootstrap (`importLibrary`). **Legacy `.pac-*` CSS must live in `base.css` with `!important`** — Google injects inline styles on `.pac-container`, so rules without `!important` are silently overridden. Do not put `.pac-*` rules in page-specific CSS files. GCP requirement: "Places API (New)" must be enabled separately from the legacy "Places API" in Google Cloud Console for both frontend and backend keys.
- **Two Google Maps API keys — frontend vs backend**: `GOOGLE_MAPS_API_KEY` is the server-side key (used by `worker.py`, `cli.py`, `property_evaluator.py` for Geocoding, Distance Matrix, Places). `GOOGLE_MAPS_FRONTEND_API_KEY` is a separate, domain-restricted key rendered into browser HTML for Maps JavaScript API + Places Autocomplete. There is **no fallback** between them — if `GOOGLE_MAPS_FRONTEND_API_KEY` is unset, the `{% if config.GOOGLE_MAPS_FRONTEND_API_KEY %}` guard suppresses the entire Maps `<script>` tag and autocomplete silently disappears with no error. When deploying to a new environment or migrating GCP projects, both keys must be set independently. The frontend key should have HTTP referrer restrictions (e.g., `nestcheck.org/*`); the backend key should have IP restrictions or no restrictions (server-to-server).

### `index.html` (near address input)
- **Password manager suppression on non-credential inputs**: Address and search inputs with `name="address"` trigger password manager autofill (1Password, LastPass, Dashlane) because they match contact/identity heuristics. Add all four attributes to suppress: `autocomplete="off"` (browser), `data-1p-ignore` (1Password), `data-lpignore="true"` (LastPass), `data-form-type="other"` (Dashlane). `autocomplete="off"` alone is insufficient — each manager requires its own vendor-specific opt-out. When adding new text inputs to templates, apply these attributes if the field is not a login/identity field. Does NOT interfere with Google Places Autocomplete, which operates at the DOM/JS level.

### Relevant CSS files (flex/grid patterns)
- **Nested flex containers need their own `gap`**: CSS `gap` only applies between direct children of a flex container. When dynamically-generated items (e.g., pill buttons) are inside a wrapper `<div>` that is itself a flex child, the parent's `gap` spaces the wrapper from siblings — not the items inside it. Always add `display: flex; gap: Npx;` to the wrapper too, or append items directly to the flex container.
- **Negative-margin background extension must respect mobile padding**: Using `margin-left: calc(-1 * var(--space-xl))` to extend a section's background to container edges only works if the parent container has at least `--space-xl` padding. When the parent padding reduces at mobile breakpoints (e.g., `--space-base` at 640px), the negative margin exceeds the padding and causes horizontal overflow. Either: (1) reset the negative margin at the breakpoint, or (2) use a different technique (e.g., pseudo-element with `position: absolute`).
- **Grid `max-width` must include padding**: When a grid container has `max-width: Npx` and `padding: 0 Mpx`, the columns compete for `N - 2M` pixels, not `N`. A `max-width: 1024px` container with 24px padding holds only 976px for columns. If the columns sum to 1024px (720 + 24 gap + 280), they won't fit. Set `max-width` to `column-sum + 2 × padding`.

### `index.html` (near cookie functions)
- **Cookie-based client state pattern**: `nestcheck_recent` cookie stores a JSON-encoded array of `{address, snapshot_id}` objects (max 3, 30-day expiry, `SameSite=Lax; Path=/`). `place_id` is only included when non-null to reduce cookie size — all consumers access it via optional patterns (`entry.place_id || null`). `getRecentSearches()` handles graceful migration from the legacy string-array format (checks `typeof parsed[0] === 'string'`). Read with regex on `document.cookie`, write with `setCookie(name, value, days)`. Always wrap `JSON.parse` in try/catch and return `[]` on failure — cookies can be corrupted by browser extensions or manual edits. The `addRecentSearch` / `getRecentSearches` / `renderRecentSearches` / `updateRecentSearchSnapshot` functions in `index.html` are the reference implementation. Pill clicks with a `snapshot_id` check `/api/snapshots/<id>/fresh` before redirecting — stale or missing snapshots fall through to a new evaluation.

### `models.py` (near `get_snapshot()`)
- **`get_snapshot()` loads full `result_json`**: The `get_snapshot()` function in `models.py` does `SELECT *` and parses `result_json` (potentially 100KB+). For lightweight checks (existence, freshness), this is wasteful. If a high-frequency endpoint only needs metadata columns (`evaluated_at`, `snapshot_id`), consider adding a `get_snapshot_metadata()` function that selects only non-blob columns. Currently `api_snapshot_fresh` uses `get_snapshot()` — acceptable for now but worth optimizing if the endpoint sees high traffic.

### `report.css` (CSS specificity patterns)
- **CSS opacity stacking on dual-classed elements**: When adding an opacity-based modifier class (e.g., `.section-card--quiet { opacity: 0.85 }`) to elements that already have an opacity rule (e.g., `.venue-card--muted { opacity: 0.75 }`), the later-declared rule wins in CSS cascade — it doesn't multiply. This can unintentionally *increase* opacity. Add a compound selector (`.venue-card--muted.section-card--quiet { opacity: 0.75 }`) to preserve the original intent.
- **`display: contents` for layout-mode-dependent wrappers**: When a wrapper div is needed in one layout mode (row) but must be invisible in another (column), `display: contents` makes the element's children participate directly in the parent's flex/grid layout as if the wrapper doesn't exist. Reference: `.venue-card__info` uses `display: contents` by default (scroll mode) and switches to `display: flex; flex-direction: column` in stack mode. Caveat: `display: contents` removes the element from the accessibility tree in Safari < 16 — only use on non-semantic wrappers.
- **`:last-child` on variable-count siblings needs `:not(:first-child)` guard**: When styling the "secondary" item in a group (e.g., secondary time pill below primary), `:last-child` also matches a solo element that is both first and last. Use `:last-child:not(:first-child)` to only target items when siblings exist. This bug is invisible during testing if all test cases have 2+ items — it only surfaces when a venue has a single time pill (walk-only, no drive).
- **Base class + modifier cascade: check for property bleed-through**: When a base CSS class (e.g., `.verdict-card`) is declared after a modifier (e.g., `.verdict-card.score-failed`), the modifier has higher specificity and wins on properties it sets — but any properties set ONLY on the base (like `box-shadow`, `border`) bleed through to the modifier variant. When adding a new base class to an element that already has modifier variants, audit all modifiers for properties that now need explicit resets (e.g., `box-shadow: none; border: none;`).

### `homepage.css` (near `#what-we-evaluate`)
- **Scroll-triggered fade-in must be JS-gated, not CSS-default**: When using `opacity: 0` + IntersectionObserver to reveal elements on scroll, the CSS `opacity: 0` rule must only apply when JS has opted in (e.g., `#el.hp-fade-ready:not(.hp-visible) { opacity: 0 }`). JS adds the gate class (`hp-fade-ready`) before observing. Without this, the element is invisible when JS is disabled or fails. The `prefers-reduced-motion` CSS block must also set `opacity: 1` as a fallback. Reference: `#what-we-evaluate` in `homepage.css` (NES-314).

### CSS token migration patterns
- **Remove `opacity` when migrating to semantic color tokens**: If a class previously had no explicit `color` (inheriting a dark parent color) and used `opacity: 0.7` to soften it, migrating to an already-faint color token (e.g., `--type-l5-color: var(--color-text-faint)`) while keeping the `opacity` produces double-faintness — 70% opacity of an already-light gray becomes near-invisible. When adding an explicit `color` that carries the intended faintness, remove the `opacity` rule.
- **CSS `var()` fallbacks are dead code when the token is defined**: `var(--color-bg, #ffffff)` looks like it defaults to white, but if `--color-bg` resolves to `#F8F9FB` (off-white), the fallback is never reached. Always verify the token's actual resolved value in `tokens.css`, not the fallback. Use `--color-bg-card` (`#FFFFFF`) for true white backgrounds on card surfaces, `--color-bg` / `--color-bg-page` (`#F8F9FB`) for page-level backgrounds. Match fallback hex values to the token's definition to avoid confusion.
- **CSS token name collisions across naming conventions**: When a token system has both bare-prefix tokens (`--band-limited: #EA580C`) and namespaced tokens (`--color-band-limited: #6B7280`), the similar names resolve to completely different values. This happens when legacy 5-band tokens coexist with spec 3-band tokens. Always verify which token you're referencing by checking the actual resolved value in `tokens.css`, not the name pattern.

### `index.html` (near crossfade code)
- **`transitionend` listeners need a fallback timer**: When using `el.style.opacity = '0'` + `transitionend` to sequence a fade-out → text swap → fade-in, the `transitionend` event may never fire (element hidden, browser skips transition, tab backgrounded). Always add a `setTimeout` fallback at slightly longer than the CSS transition duration (e.g., 250ms for a 200ms transition). The `transitionend` handler clears the fallback with `clearTimeout`; if it never fires, the fallback performs the same update. Without this, the element stays at `opacity: 0` permanently. Reference: loading overlay crossfade in `index.html`.

### Various CSS files (color/background patterns)
- **Semi-transparent backgrounds depend on container color**: `rgba(255, 255, 255, 0.8)` is visible on a yellow background but invisible on white. When changing a container's background color, audit child elements for semi-transparent backgrounds that relied on contrast with the old color. Replace with opaque token values (e.g., `var(--color-border-light)`) that work regardless of the container.

### NLCD Tree Canopy Cover (canopy.py, NES-347) → `canopy.py` module docstring
- **Standalone module**: `canopy.py` queries MRLC WMS endpoint for NLCD 30m tree canopy data. No Flask, no Google API dependencies — only `requests` + `models.py` cache helpers. Follows `green_space.py`/`overflow.py` standalone pattern.
- **MRLC WMS endpoint**: `https://www.mrlc.gov/geoserver/mrlc_display/nlcd_tcc_conus_2021_v2021-4/wms` — free, no auth, CONUS coverage. Use WMS 1.1.1 (not 1.3.0) to avoid lat/lng axis order confusion. `GetFeatureInfo` returns `PALETTE_INDEX` (0-100 = canopy %).
- **Grid sampling**: `_generate_sample_grid()` produces ~21 points in a 5x5 grid within 500m buffer, pruned to circle. Uses latitude-corrected longitude spacing (`cos(radians(lat))`). The 0.8× buffer scaling ensures grid points fit within the circle (corners of a 1× grid land at ~1.13× buffer). The 1.02× tolerance on the distance check absorbs floating-point imprecision at high latitudes.
- **25 parallel WMS requests**: `ThreadPoolExecutor(max_workers=5)` parallelizes the HTTP GETs. Adds ~3-5 seconds to uncached evaluations. Cached results return instantly (90-day TTL).
- **Scoring integration**: Canopy % replaces the keyword-based `_score_nature_feel()` when available — no double-counting. `apply_piecewise(CANOPY_NATURE_FEEL_KNOTS, canopy_pct)` maps 0-100% to 0-2 nature-feel subscore. When canopy data is unavailable (endpoint down, outside CONUS), falls back to OSM tags + name keywords unchanged.
- **Data flow**: `evaluate_property()` → `_staged("canopy", get_canopy_cover, lat, lng)` → `result.canopy_cover` (dict) → `evaluate_green_escape(canopy_pct=...)` → each `score_green_space(canopy_pct=...)`. Serialized in `result_to_dict()` under `green_escape.canopy_cover`. Template displays via `result.green_escape.canopy_cover`.
- **Coverage**: `NLCD_CANOPY` in `coverage_config.py` as a live API source (`table: None`, `state_filter: None`), active for all 8 states.

### Green Space Scoring (green_space.py) → `green_space.py` near `_apply_walk_time_ceiling()`
- **Walk-time ceiling on park Daily Value** (NES-391): `_apply_walk_time_ceiling()` in `green_space.py` caps the total Daily Value score based on walk time, applied after subscore summation but before the `min(10.0, total)` cap. Graduated tiers: ≤15 min uncapped, 16-20 cap 8, 21-25 cap 6, 26-30 cap 5, >30 cap 3. Applied in both `compute_park_score()` (pure function) and `score_green_space()` (API-calling function). `_evaluate_criteria()` receives the ceilinged total — intentional, so criteria classification reflects final accessible-park value. Constants: `WALK_CEILING_TIERS` and `WALK_CEILING_BEYOND_MARGINAL` in `green_space.py`.

### Empty State Copy Library (copy_library.py, NES-319) → `copy_library.py` module docstring

### Narrative Insights (NES-191+) → module-level docstring in insight generator modules
- Insight generators are pure functions: `dict → str | None`. Keep them side-effect-free for testability.
- Place insight logic in dedicated modules (not `app.py`) — `app.py` is for routes/views.
- **Dict field-name contracts**: Insight functions that consume dicts from `present_checks()` must use the field names that function actually produces (`"name"`, `"result_type"`, `"category"`, etc.) — not alternative names like `"display_name"` or `"check_id"`. When adding a new consumer of presented-check dicts, verify field names against `present_checks()` output, not against `_PROXIMITY_LABELS` keys or other lookup tables. Always add a `or "this hazard"` (or similar) terminal fallback when interpolating dict values into user-facing sentences to prevent sentence fragments from malformed input.
- Avoid duplicating utility helpers across modules (e.g., Oxford-comma join) — use a shared module.
- **`_PROXIMITY_LABELS` must use "a/an + singular noun" pattern**: Every entry is interpolated into sentences like `"This address is close to {label}"` and joined with `_join_and()`. Plural or article-free entries (e.g., `"high-voltage power lines"` instead of `"a high-voltage power line"`) produce awkward grammar when combined with other labels that do have articles.
- **Test assertions on prose should be semantic, not literal**: Insight generator tests that assert on exact prose fragments (`"driving"`, `"on the other hand"`, `"no parks"`) break during content design passes. Prefer assertions on structural properties (entity names present, `car_dependent` flag, key semantic words like `"car"`) over verbatim copy. When a content pass updates N prose strings, expect ~N test assertion updates — budget for it.
- **HTML-escape external data in `| safe` narrative generators**: Functions that return HTML strings rendered with Jinja's `| safe` filter (e.g., `generate_report_narrative`, any future insight generator returning HTML) must escape all external data (place names from Google Places API, user input, check headlines) via `markupsafe.escape()` before interpolation. Without this, a malicious place name like `<script>alert(1)</script>` executes in the browser. Pattern: `str(_html_escape(name))` at the point of extraction, not at interpolation.
- **`_join_labels()` requires a non-empty list**: The function has no internal guard for empty lists — `_join_labels([])` crashes with `IndexError`. When the input list is built from a filter that can reduce to zero elements (e.g., `[i for i in items if i]`), always guard the call: `if filtered: _join_labels(filtered)`. This is easy to miss when the upstream list is truthy but contains only falsy elements like empty strings.

### SQLite Concurrency (models.py) → `models.py` near `_get_db()`
- **`nestcheck.db` path does NOT auto-detect Railway volume**: `models.py` uses `os.environ.get("NESTCHECK_DB_PATH", "nestcheck.db")` — defaulting to the ephemeral working directory. In contrast, `spatial_data.py` auto-detects via `RAILWAY_VOLUME_MOUNT_PATH`. If `NESTCHECK_DB_PATH` is not set in Railway's dashboard (pointing to `/app/data/nestcheck.db`), every deploy wipes the operational database (jobs, snapshots, payments). This asymmetry caused a "Job not found" failure when a deploy swapped the container mid-evaluation — the new container had a fresh DB. Verify `NESTCHECK_DB_PATH=/app/data/nestcheck.db` is set in the Railway dashboard. Future fix: add `RAILWAY_VOLUME_MOUNT_PATH` auto-detection to `models.py` matching the `spatial_data.py` pattern.
- `_get_db()` sets `PRAGMA busy_timeout=30000` (30s) for write contention under concurrent gunicorn workers + evaluation threads.
- Job-critical write functions (`create_job`) retry up to 3× on `OperationalError` with "locked"/"busy" in the message, with progressive backoff. Read functions (`get_job`) retry once.
- Pattern for retry-safe DB functions: open connection inside `try`, use `finally: conn.close()`, catch `sqlite3.OperationalError` outside the connection block.
- `sqlite3.connect(timeout=N)` and `PRAGMA busy_timeout` both set the busy timeout — use only the PRAGMA (more explicit, avoids redundancy).

### Data Caches (models.py) → `models.py` near cache functions
- Pattern: `get_<name>_cache(key) → Optional[str]` / `set_<name>_cache(key, json) → None`
- Always swallow cache errors so they never break an evaluation.
- Each cache table has its own TTL constant (`_<NAME>_CACHE_TTL_DAYS`).
- Existing caches: weather (30-day), census (90-day), canopy (90-day).

The helper returns typed data structures (health grid, dimension rows, key differences) — the template only renders. This keeps business logic testable and templates simple.

### Automation Scripts (scripts/) → script-level docstrings
- **`result_to_dict()` vs `view_snapshot()` field contracts**: `result_to_dict()` produces the core evaluation dict (coordinates, tier1_checks, tier2_scores, final_score, neighborhood_places, etc.). Additional fields like `presented_checks`, `score_band`, `verdict`, `dimension_summaries`, `structured_summary`, and `walkability_summary` are added by `view_snapshot()` or other route handlers downstream. Scripts that validate result structure must validate against the contract of the function they actually call — `result_to_dict()` for evaluator-level validation, or the full snapshot dict for end-to-end validation. Validating `result_to_dict()` output against `view_snapshot()` keys produces false failures.
- **Railway volumes cannot be shared — use HTTP endpoints**: Each Railway service can only mount its own volume. The workaround for cron services that need production data is an authenticated HTTP endpoint on the web service. The spatial-health cron calls `GET /api/spatial-health` on the web service (authenticated via `SPATIAL_HEALTH_TOKEN`) instead of running its own ingestion. This pattern can be reused for future cron→web service communication.
- **Railway cron config is per-service via separate `.toml` files**: `railway.toml` configures the web service. Each cron service points to its own `railway-cron-*.toml` file via the "Railway Config File" dashboard setting. All three cron `.toml` files use `restartPolicyType = "NEVER"` since cron services must exit after completing their task.
- **Cross-script imports in `scripts/` need path setup**: Scripts in `scripts/` add `_PROJECT_ROOT` to `sys.path`, which enables imports from the project root (`property_evaluator`, `app`, etc.) but not from `scripts/` itself. `from regression_baseline import TEST_ADDRESSES` in `daily_smoke_test.py` works because Python adds the script's own directory to `sys.path` when run as `__main__`, but this breaks if the module is imported from elsewhere. Prefer inlining shared constants or extracting to a project-root module.
- **Importing from `scripts/` in app.py requires `scripts/__init__.py`**: `from scripts.spatial_health_check import check_health` needs `scripts/` to be a Python package. Without `__init__.py`, the import raises `ModuleNotFoundError` at runtime — easy to miss since it's a lazy import inside a route handler, not a startup import. The `__init__.py` exists (empty) specifically for this purpose. When adding new cross-boundary imports from `scripts/` into app code, verify the package file exists.
- **HTTP endpoints returning error status codes still carry valid data**: `_fetch_from_endpoint()` must handle 503 as valid JSON (unhealthy data), not as a network error. `urllib` raises `HTTPError` for non-2xx — catch it, read the body, and parse. Always guard `json.loads()` with `JSONDecodeError` handling when consuming HTTP responses, since proxies and load balancers can inject HTML error pages at any status code.

### Railway Cron Services (`railway-cron-*.toml`) → `railway-cron-*.toml` headers or `docs/railway-cron-setup.md`
- **`railway.toml` is single-service scoped**: It configures only the web service. Railway cannot define multiple services in one config file. Each cron service points to its own `railway-cron-<name>.toml` via the dashboard's "Config File Path" setting.
- **Railway volumes cannot be shared — HTTP endpoint workaround**: The web service owns `/app/data` with `spatial.db`. Cron services that need spatial data use an authenticated HTTP endpoint on the web service instead of their own volume. The spatial-health cron calls `GET /api/spatial-health` (Bearer token via `SPATIAL_HEALTH_TOKEN`) which checks the production volume directly. Env vars: `SPATIAL_HEALTH_TOKEN` (shared secret, set on both web and cron services), `NESTCHECK_WEB_URL` (cron-only, production URL).
- **Cron services must exit**: Railway skips subsequent cron executions if the previous one is still running. All automation scripts (`daily_smoke_test.py`, `spatial_health_check.py`, `regression_baseline.py`) are designed to exit 0 (success) or 1 (failure). Never add long-running loops or retry-forever patterns.
- **`restartPolicyType = "NEVER"` for cron**: Cron services should not auto-restart on failure — the next scheduled run handles retry. Only the web service uses `ON_FAILURE`.
- **Env var expansion in `startCommand`**: Shell variables like `$SMOKE_TEST_NOTIFY_EMAIL` in `startCommand` expand to empty string if unset, which passes `--email ""` to the script. Scripts should guard against empty-string email args (use `args.email or os.environ.get(...)` pattern).
- **Cross-script imports need `scripts/` on `sys.path`**: Scripts in `scripts/` that import from other scripts in the same directory (e.g., `from regression_baseline import TEST_ADDRESSES`) must add `_SCRIPT_DIR` to `sys.path`, not just `_PROJECT_ROOT`. The project root only covers top-level modules like `property_evaluator` and `app`. Pattern: `sys.path.insert(0, str(_PROJECT_ROOT))` then `sys.path.insert(0, str(_SCRIPT_DIR))`.
- **Three automation scripts and their Makefile targets**: `make smoke-test` (daily pipeline check), `make spatial-health` (weekly data integrity), `make regression` (monthly scoring stability). Email variants: `make smoke-test-email`, `make spatial-health-email`. Record baselines: `make spatial-baseline`, `make regression-update`.

### Mobile Tab Bar + Section Navigation (NES-323) → `section-nav.js` + `_tab_bar.html`
- **`section-nav.js` uses a registration pattern for nav consumers** (NES-328): `SectionNav.register({ containerSelector, linkSelector, activeClass, onActivate })` adds a nav surface. The single `IntersectionObserver` calls all registered callbacks with the active section ID — it doesn't know which surfaces exist. Built-in consumers (desktop rail `.report-rail-nav`, mobile tab bar `.mobile-tab-bar`) are registered in the bootstrap IIFE at file bottom. To add a third nav surface, add one `SectionNav.register()` call — no changes to the observer. When adding a new report section, add the `data-section` link to BOTH `_report_rail.html` AND `_tab_bar.html`. Smooth-scroll-on-click uses `window.scrollTo` with a 52px top offset (`SCROLL_OFFSET` constant) so the target clears the sticky tab bar — never use `scrollIntoView()` for this because it ignores fixed/sticky headers.
- **Mobile tab order intentionally differs from desktop rail**: The PRD §6.2 specifies Verdict → Health → Dimensions on mobile (Health 2nd), while the desktop rail has Verdict → Dimensions → Health (Dimensions 2nd). Do not "fix" this — it's by design.
- **`_tab_bar.html` must be included in ALL templates that use `.report-layout`**: Currently `snapshot.html` and `index.html`. The tab bar is `display: none` at >= 1024px and sticky at `top: 0` below 1024px.
- **Never use `element.scrollIntoView()` to scroll within an `overflow-x: auto` container**: It can cause unwanted vertical page scroll on some browsers (Safari). Instead, use `container.scrollTo({ left: ..., behavior: "smooth" })` with manual offset calculation. The tab bar's active-tab centering uses this pattern.
