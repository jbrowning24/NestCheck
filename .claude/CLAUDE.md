# NestCheck

Property evaluation tool for Westchester County rentals. Analyzes health, lifestyle, and budget criteria for Metro-North corridor properties.

## Tech Stack

- **Backend:** Python/Flask
- **Database:** SQLite with async evaluation queue
- **APIs:** Google Maps, Overpass (OpenStreetMap)
- **Frontend:** Jinja templates, vanilla JS
- **Hosting:** Railway

## Project Structure

```
NestCheck/
├── app.py              # Flask routes, API endpoints

├── nc_trace.py         # Request tracing

```

## Claude Code Configuration

### Custom Commands & Hooks
- Custom slash commands in `.claude/commands/` must not shadow built-in commands (e.g., `/review`, `/compact`, `/init`). Check the built-in list before naming a new command. Shadowed built-ins become unreachable.
- PostToolUse hooks that reference `$CLAUDE_FILE_PATH` must guard with `test -n` and quote the variable. If unset, tools like `ruff format` will silently operate on the entire directory.
- Subagent definitions live in `.claude/agents/`. Each agent should have a single focused responsibility and explicit verification steps.
- SessionStart hooks that run `sudo apt install` will hang in environments requiring a password prompt. Always guard with `command -v <tool>` to skip installation when the tool already exists, and end with `|| true` so failures never block the session.

## Coding Standards

- Python: Follow existing patterns in property_evaluator.py
- Use type hints for function signatures
- Add docstrings for public functions
- No print() in production - use logging
- All API calls need timeout handling
- **Never use predictable defaults for secrets**: `os.environ.get("SECRET", "default-value")` is a security vulnerability if the default is guessable and the env var isn't set in production. For `SECRET_KEY`, `BUILDER_SECRET`, and similar auth/session secrets: use `os.environ.get("VAR")` (returns `None`) and guard all usage with `if VAR and ...`. For `SECRET_KEY` specifically, fail fast if it's the dev default and `RAILWAY_ENVIRONMENT` is set.
- **SQL table name interpolation must use whitelists**: f-string table names in SQLite (e.g., `f"SELECT ... FROM {table_name}"`) cannot be parameterized with `?`. Always validate against a whitelist before interpolation. In `spatial_data.py`, `_VALID_FACILITY_TYPES` is the canonical whitelist; `_validate_facility_type()` returns `None` for unknown types (graceful degradation in queries) or the validated table name. When adding a new spatial dataset, add its facility_type to `_VALID_FACILITY_TYPES` in the same commit as the ingest script.
- **Never store raw `str(e)` in user-visible DB fields**: Exception messages can contain file paths, SQL queries, API keys from error URLs, or internal stack details. Use `_sanitize_error(e)` in `worker.py` to classify exceptions into safe user-facing messages. Full details go to `logger.exception` only. The sanitizer matches on keyword presence (`"timeout"`, `"api"`, `"geocod"`, `"config"`) and returns one of five hardcoded strings.
- **Unbounded class-level caches need eviction**: Any `Dict` or `dict` used as a process-lifetime cache (shared across instances via class variables) must have a size bound. Use `OrderedDict` + `move_to_end()`/`popitem(last=False)` for LRU, or `functools.lru_cache` for function-level caching. `urban_access.py`'s `_cache_set()` is the reference pattern (1000-entry LRU with `threading.Lock`).
- Spatial metadata values may arrive as strings after JSON round-tripping — cast to `float()` before numeric formatting (`:,.0f`)
- **scoring_config ↔ property_evaluator import sync**: When adding new constants to `scoring_config.py` that are referenced in `property_evaluator.py`, update the `from scoring_config import (...)` block in the same commit. Missing imports surface as `NameError` only at evaluation runtime (inside try/except), not at startup — so they silently produce 0/10 scores with error text in reports instead of crashing visibly.
- When changing template element IDs, update `smoke_test.py` markers (`LANDING_REQUIRED_MARKERS`, `SNAPSHOT_REQUIRED_MARKERS`) in the same commit. Mismatches cause silent post-deploy smoke test failures.
- When removing HTML elements from templates, remove the corresponding CSS rules in the same commit. Orphaned selectors (e.g., `.snippet-assessment-score` after removing the score div) accumulate silently.
- When promoting a CSS pattern from a page-specific file to `base.css` (e.g., sticky footer), remove the now-redundant rules from page-specific files (e.g., `error-page.css`) in the same commit. Since page-specific files load after `base.css`, the duplicates are silently harmless but accumulate as confusing dead code.
- **When inlining logic that replaces helper functions, remove the old helpers in the same commit.** JS helper functions (`stageDisplay()`, `stagePercent()`) left behind after their call sites are refactored to inline lookups are dead code that confuses future readers. Grep for the function name after refactoring to confirm zero remaining call sites before deleting.
- **Page-scoped CSS class naming**: Generic class names in page-specific stylesheets (e.g., `.section-label` in `pricing.css`) can collide with identically-named rules in globally-loaded stylesheets (e.g., `report.css`). If a page-specific class name is generic, prefix it with the page name (`.pricing-section-label`) to prevent silent style bleed if stylesheets are ever co-loaded.
- **External links always need `rel="noopener noreferrer"`**: `noreferrer` implies `noopener` and additionally suppresses the `Referer` header. Use both on every `target="_blank"` link — `rel="noopener"` alone is incomplete.
- **Inline SVG icons over HTML entities**: Use inline SVGs (e.g., `<svg width="12" height="12" viewBox="0 0 24 24" ...><polyline .../></svg>`) instead of HTML character entities (`&#9654;`, `&#9888;`, `&#9650;`) for icons. SVGs render crisply at all sizes, inherit `currentColor`, and animate smoothly via CSS `transform`. When replacing entities with SVGs inside an existing `.collapse-icon` span, switch the span from `display: inline-block` to `display: inline-flex` and add `.collapse-icon svg { display: block }` to eliminate baseline gaps. When replacing icons across the codebase, audit ALL templates in one pass — `_result_sections.html`, `compare.html`, `compare_health.html`, `_eval_snippet.html` — to avoid inconsistent rendering between pages.
- **Score display format consistency**: All `/10` scores in the report (dimension scores, park Daily Value, park subscores, green space list scores) should display as integers (`%.0f`). Google Places star ratings (`4.5 ★`) keep one decimal (`%.1f`) since that's their natural precision. Mixing formats (e.g., "7.6/10" next to "8/10") looks like a bug.
- **New visual sub-palette vs. token reuse**: When a page intentionally introduces a neutral/monochrome sub-palette (e.g., `#111111`, `#666666`) that diverges from the site's blue-tinted token system, use tokens for *structural* values (spacing, font weights, border-radius, transitions) but leave palette colors as explicit values rather than mapping them to semantically mismatched tokens. Forcing `#111111` into `--color-text-primary: #0F172A` (navy) or `#666666` into `--color-text-muted: #64748b` (cool blue-gray) produces subtle hue drift that undermines the design intent. Within the sub-palette, consolidate similar grays (e.g., `#555555` and `#8E8E8E` both → `#666666`) to avoid the "narrow band of grays" AI-slop signal.
- **No hover effects on non-interactive elements**: Cards, scorecards, and preview panels that are display-only (no `<a>`, no click handler) must not have `:hover` styles (translateY, shadow escalation, border-color changes). Hover lifts on static content is a top AI-slop pattern — it signals interactivity that doesn't exist and makes the page feel template-generated. Only add hover transitions to genuinely clickable elements.
- **Use `color` tokens for text hierarchy, not `opacity`**: `opacity` dims the entire element — text, borders, backgrounds, shadows, child elements — and doesn't compose predictably when multiple opacity rules overlap (see CSS opacity stacking entry below). For text de-emphasis, use explicit color tokens (`--color-text-muted`, `--color-text-faint`, `--color-text-secondary`). Reserve `opacity` for overlays, disabled states, icon badges, and elements where dimming *everything* is the actual intent.
- When relocating template sections to a different parent container, update surrounding copy (section headings, divider descriptions, context notes) to reflect the new contents. Stale copy (e.g., "Community and school data" after adding EJScreen) silently misleads users.
- Frontend `fetch()` calls that expect JSON must check `resp.ok` and content-type before calling `.json()`. Non-JSON error responses (CSRF 400, HTML 500) cause Safari-specific `TypeError` ("The string did not match the expected pattern") that hides the real error. Always guard: `if (!resp.ok) { /* handle non-JSON */ }` before `resp.json()`.
- Frontend polling loops must never give up on a single transient error (404, 5xx, network). Use retry counters with a cap before showing a failure. The job queue is eventually-consistent under load; a freshly-created job may not be visible to the poll endpoint for 1-2 cycles.
- Flask error handlers (`@app.errorhandler`) must return JSON when `_wants_json()` is true. Without this, JS clients get HTML error pages they can't parse. Add handlers for 400, 404, and 500 at minimum.
- **CSRF auto-retry pattern**: `csrfFetch(url, options)` in `index.html` wraps `fetch()` with CSRF token injection, `resp.ok`/content-type guarding, and single auto-retry on `error_code === 'csrf_expired'`. Use `csrfFetch` for all new POST endpoints instead of raw `fetch` + manual `X-CSRFToken` header. The server-side `@app.errorhandler(400)` adds `"error_code": "csrf_expired"` to JSON responses when the error message contains "csrf" — client matches on that structured code, not on substring search of the human-readable error text.
- **Google Maps: PlaceAutocompleteElement (New API)**: Both `index.html` and `compare_health.html` use the Google Dynamic Library Import bootstrap (minified IIFE in `<head>`) and `google.maps.importLibrary('places')` to load the Places library on demand. Autocomplete uses `new google.maps.places.PlaceAutocompleteElement({ includedRegionCodes: ['us'], types: ['address'] })` — a web component with shadow DOM that replaces the old `google.maps.places.Autocomplete` widget. Key differences from legacy: (1) `place.id` replaces `place.place_id`, (2) `gmp-select` event replaces `place_changed`, (3) `fetchFields()` is required after selection to get place data, (4) the element manages its own dropdown styling via shadow DOM — no `.pac-*` CSS needed, (5) users must select from the dropdown — typed text lives in the shadow DOM input, not in the hidden `#address-input`. The hidden `<input name="address" id="address-input">` stays in the DOM (`display: none`) and is populated by the `gmp-select` handler so form submission and `submitEvaluation()` work unchanged. GCP requirement: "Places API (New)" must be enabled separately from the legacy "Places API" in Google Cloud Console for both frontend and backend keys. **Styling requires `className`, not inline styles**: The element's shadow DOM `::part()` selectors (input color, font, predictions dropdown, icon visibility) only match when the host element has the `.nestcheck-autocomplete` class. Setting inline styles (`style.flex`, `style.border`, etc.) instead of applying the class leaves all `::part()` CSS as dead code — the element renders with Google's default Material Design theme. Both `index.html` and `compare_health.html` create `PlaceAutocompleteElement` — apply the class in both. The autocomplete CSS currently lives in `index.css`; `compare_health.html` loads `compare.css` instead, so the rules must be moved to a shared file (e.g., `base.css`) or loaded explicitly for both pages.
- **Two Google Maps API keys — frontend vs backend**: `GOOGLE_MAPS_API_KEY` is the server-side key (used by `worker.py`, `cli.py`, `property_evaluator.py` for Geocoding, Distance Matrix, Places). `GOOGLE_MAPS_FRONTEND_API_KEY` is a separate, domain-restricted key rendered into browser HTML for Maps JavaScript API + Places Autocomplete. There is **no fallback** between them — if `GOOGLE_MAPS_FRONTEND_API_KEY` is unset, the `{% if config.GOOGLE_MAPS_FRONTEND_API_KEY %}` guard suppresses the entire Maps `<script>` tag and autocomplete silently disappears with no error. When deploying to a new environment or migrating GCP projects, both keys must be set independently. The frontend key should have HTTP referrer restrictions (e.g., `nestcheck.org/*`); the backend key should have IP restrictions or no restrictions (server-to-server).
- **Removing env var fallbacks requires deployment coordination**: When an env var assignment changes from `A or B` to just `A` (removing a fallback), the feature silently breaks in any environment where only `B` was set — and `{% if %}` template guards make the failure invisible (no error, no log, feature just vanishes). Before merging a fallback removal: (1) verify `A` is set in all deployment environments, (2) add a startup `logger.warning` if `A` is unset and the feature is expected, (3) update README/`.env.example`/deployment docs in the same commit. This applies to any env var, not just API keys.
- **Local commits on `main` get overwritten by PR merges**: When working directly on `main` with uncommitted or unpushed changes, a merged PR that touches the same file will overwrite local edits on the next pull. The NES-312 startup warning was added locally, then lost when PR #222 merged and overwrote `app.py`. For changes that must survive concurrent PR merges, either: (a) push immediately to `main` before the PR merges, or (b) work on a branch and open a separate PR.
- **Password manager suppression on non-credential inputs**: Address and search inputs with `name="address"` trigger password manager autofill (1Password, LastPass, Dashlane) because they match contact/identity heuristics. Add all four attributes to suppress: `autocomplete="off"` (browser), `data-1p-ignore` (1Password), `data-lpignore="true"` (LastPass), `data-form-type="other"` (Dashlane). `autocomplete="off"` alone is insufficient — each manager requires its own vendor-specific opt-out. When adding new text inputs to templates, apply these attributes if the field is not a login/identity field. Does NOT interfere with Google Places Autocomplete, which operates at the DOM/JS level.
- **Never use synthetic DOM events to trigger existing handlers**: `dispatchEvent(new Event('submit'))` creates a non-cancelable event that bypasses HTML5 validation and behaves differently from real user events. Instead, call the handler function directly (e.g., `submitEvaluation(addr, null, email)`) — it's clearer, more robust, and avoids subtle bugs with `preventDefault()` being a no-op on non-cancelable events.
- **Nested flex containers need their own `gap`**: CSS `gap` only applies between direct children of a flex container. When dynamically-generated items (e.g., pill buttons) are inside a wrapper `<div>` that is itself a flex child, the parent's `gap` spaces the wrapper from siblings — not the items inside it. Always add `display: flex; gap: Npx;` to the wrapper too, or append items directly to the flex container.
- **Cookie-based client state pattern**: `nestcheck_recent` cookie stores a JSON-encoded array of `{address, snapshot_id, place_id}` objects (max 5, 30-day expiry, `SameSite=Lax; Path=/`). `getRecentSearches()` handles graceful migration from the legacy string-array format (checks `typeof parsed[0] === 'string'`). Read with regex on `document.cookie`, write with `setCookie(name, value, days)`. Always wrap `JSON.parse` in try/catch and return `[]` on failure — cookies can be corrupted by browser extensions or manual edits. The `addRecentSearch` / `getRecentSearches` / `renderRecentSearches` / `updateRecentSearchSnapshot` functions in `index.html` are the reference implementation. Pill clicks with a `snapshot_id` check `/api/snapshots/<id>/fresh` before redirecting — stale or missing snapshots fall through to a new evaluation.
- **`get_snapshot()` loads full `result_json`**: The `get_snapshot()` function in `models.py` does `SELECT *` and parses `result_json` (potentially 100KB+). For lightweight checks (existence, freshness), this is wasteful. If a high-frequency endpoint only needs metadata columns (`evaluated_at`, `snapshot_id`), consider adding a `get_snapshot_metadata()` function that selects only non-blob columns. Currently `api_snapshot_fresh` uses `get_snapshot()` — acceptable for now but worth optimizing if the endpoint sees high traffic.
- **Template ↔ backend data contract verification**: Before adding Jinja2 template code that accesses a new field (e.g., `dim.band`, `result.neighborhood_summary`), verify the field is actually populated in the backend (`result_to_dict()` in `app.py` or the evaluator). Jinja2's `is defined` guards and `|default()` filters silently hide missing fields — the template renders without error but the feature is inert. Always grep the backend for the field name before writing template code that consumes it. This class of bug is invisible in smoke tests and only surfaces during visual QA.
- **CSS opacity stacking on dual-classed elements**: When adding an opacity-based modifier class (e.g., `.section-card--quiet { opacity: 0.85 }`) to elements that already have an opacity rule (e.g., `.venue-card--muted { opacity: 0.75 }`), the later-declared rule wins in CSS cascade — it doesn't multiply. This can unintentionally *increase* opacity. Add a compound selector (`.venue-card--muted.section-card--quiet { opacity: 0.75 }`) to preserve the original intent.
- **Scroll-triggered fade-in must be JS-gated, not CSS-default**: When using `opacity: 0` + IntersectionObserver to reveal elements on scroll, the CSS `opacity: 0` rule must only apply when JS has opted in (e.g., `#el.hp-fade-ready:not(.hp-visible) { opacity: 0 }`). JS adds the gate class (`hp-fade-ready`) before observing. Without this, the element is invisible when JS is disabled or fails. The `prefers-reduced-motion` CSS block must also set `opacity: 1` as a fallback. Reference: `#what-we-evaluate` in `homepage.css` (NES-314).
- **Remove `opacity` when migrating to semantic color tokens**: If a class previously had no explicit `color` (inheriting a dark parent color) and used `opacity: 0.7` to soften it, migrating to an already-faint color token (e.g., `--type-l5-color: var(--color-text-faint)`) while keeping the `opacity` produces double-faintness — 70% opacity of an already-light gray becomes near-invisible. When adding an explicit `color` that carries the intended faintness, remove the `opacity` rule.
- **CSS `var()` fallbacks are dead code when the token is defined**: `var(--color-bg, #ffffff)` looks like it defaults to white, but if `--color-bg` resolves to `#F8F9FB` (off-white), the fallback is never reached. Always verify the token's actual resolved value in `tokens.css`, not the fallback. Use `--color-bg-card` (`#FFFFFF`) for true white backgrounds on card surfaces, `--color-bg` / `--color-bg-page` (`#F8F9FB`) for page-level backgrounds. Match fallback hex values to the token's definition to avoid confusion.
- **`transitionend` listeners need a fallback timer**: When using `el.style.opacity = '0'` + `transitionend` to sequence a fade-out → text swap → fade-in, the `transitionend` event may never fire (element hidden, browser skips transition, tab backgrounded). Always add a `setTimeout` fallback at slightly longer than the CSS transition duration (e.g., 250ms for a 200ms transition). The `transitionend` handler clears the fallback with `clearTimeout`; if it never fires, the fallback performs the same update. Without this, the element stays at `opacity: 0` permanently. Reference: loading overlay crossfade in `index.html`.
- **Semi-transparent backgrounds depend on container color**: `rgba(255, 255, 255, 0.8)` is visible on a yellow background but invisible on white. When changing a container's background color, audit child elements for semi-transparent backgrounds that relied on contrast with the old color. Replace with opaque token values (e.g., `var(--color-border-light)`) that work regardless of the container.
- **Button classes on mixed elements**: `.share-btn` is used on both `<button>` and `<a>` elements. The base class includes `text-decoration: none` and `color: var(--color-text-inverse)` to ensure anchors look identical to buttons. When promoting dropdown menu items to direct buttons, always check if the target element type changes (`<button>` in menu → `<a>` for download links) and ensure the shared class handles both.
- **Nav links are defined once in `_base.html`** (NES-296): Global nav links (Pricing, Coverage, auth) live in `_base.html` only. The `nav_links` block is reserved for genuinely page-specific additions (e.g., "Evaluate an address" CTA on snapshot/compare, builder Dashboard). When adding a new site-wide nav link, add it to `_base.html` with `aria-current="page"` handling (`{% if request.path == '/route' %} aria-current="page"{% endif %}`). Do NOT add it to individual templates' `nav_links` blocks. CSS: `nav a.nav-link[aria-current="page"]` provides the active-state style.
- **Canonical logo is pure CSS in `base.css`**: The NestCheck logo mark (two skewed rectangles) is rendered with `nav .logo-icon span` elements — not an SVG or image. The logo lives exclusively in the `<nav>` in `_base.html`; do not duplicate it in report headers or other components. The icon uses `var(--color-brand)` (#0B1D3A) with the second rectangle at `opacity: 0.6`. Since the nav has a white background, the dark rectangles need no color override. If the nav background ever changes back to dark, the logo icon colors must be updated to maintain contrast.
- **Favicon mirrors the CSS logo mark**: `static/images/favicon.svg` (primary) and `static/images/favicon-32x32.png` (fallback for Safari < 15) reproduce the two-skewed-rectangles logo at 32×32. Both `<link rel="icon">` tags are in `_base.html` `<head>`. When changing the logo's brand color or geometry, update both favicon files in the same commit.
- **Canonical domain redirect with exempt paths**: `_redirect_to_canonical()` `before_request` handler 301-redirects non-canonical hosts (e.g., Railway URL) to `nestcheck.org`. `_CANONICAL_EXEMPT_PATHS` frozenset exempts paths that must work on any domain: `/healthz` (Railway health probes), `/api/spatial-health` (cron service), `/webhook/stripe` (Stripe POST), `/robots.txt` (crawlers). When adding a new endpoint that receives traffic from infrastructure (webhooks, health probes, cron callbacks), add its path to `_CANONICAL_EXEMPT_PATHS`.
- **Base class + modifier cascade: check for property bleed-through**: When a base CSS class (e.g., `.verdict-card`) is declared after a modifier (e.g., `.verdict-card.score-failed`), the modifier has higher specificity and wins on properties it sets — but any properties set ONLY on the base (like `box-shadow`, `border`) bleed through to the modifier variant. When adding a new base class to an element that already has modifier variants, audit all modifiers for properties that now need explicit resets (e.g., `box-shadow: none; border: none;`).
- **Jinja2 macros that use `| safe` must document the XSS contract**: Macros like `callout(text)` and `data_row(icon)` render parameters with `| safe` to support inline HTML (SVG icons, links). The safety contract is: hardcoded template strings are fine; any external data (API results, user input) MUST be escaped via `markupsafe.escape()` before being passed to the macro. Document this in the macro's docstring comment. This extends the existing `generate_report_narrative` rule to all `| safe` macro parameters.
- **Jinja2 cannot import underscore-prefixed macros**: `{% from "_macros.html" import _my_helper %}` raises `TemplateAssertionError`. This is a Jinja2 security restriction — names starting with `_` are considered private. When a helper macro is only used internally by other macros in the same file, it still needs a public name (e.g., `a_an` not `_a_an`). The underscore convention from Python does not carry over.
- **`dimension_details_data` pattern for presentation-layer fields** (NES-315): When scoring functions need to pass structured data to templates without widening `Tier2Score` (which would touch serialization, compare, CSV export paths), return a `details_data` dict alongside the existing return value. Store on `EvaluationResult.dimension_details_data` keyed by dimension name. `result_to_dict()` merges into `dimension_summaries`. For functions whose return type shouldn't change (many test call sites), build `details_data` in `evaluate_property()` from existing data instead. This keeps presentation concerns out of the data model.
- **Avoid `*:hover` universal selectors in CSS**: `*:hover > .some-class` forces hover-state evaluation on every element in the DOM. On report pages with hundreds of elements, this causes unnecessary style recalculation. Scope hover selectors to the actual parent elements that will be hovered (e.g., `a:hover .icon`, `.collapsible-toggle:hover .icon`).
- **Sticky sidebar scroll containment**: Any `position: sticky` sidebar/rail must have `max-height: calc(100dvh - <top>px)` + `overflow-y: auto` + `overflow-x: hidden` + `min-width: 0`. Without max-height, tall content extends beyond the viewport with no independent scroll. Without `min-width: 0`, CSS grid children use `min-width: auto` which prevents shrinking below content size, causing page-level horizontal scroll. On mobile breakpoints where the sidebar becomes `position: static`, reset with `max-height: none; overflow-y: visible`. Reference: `.report-sidebar` in `report.css`, `.report-layout__rail` in `snapshot.css`.
- **Token replacement on dark backgrounds must check contrast**: When replacing hardcoded colors with token references (e.g., `#e0e0e0` → `var(--color-text-tertiary)`), verify the token's resolved value has sufficient contrast against the element's background. Token names describe semantic *role* on the primary surface (white/light), not *luminance*. `--color-text-tertiary` (#94A3B8) is readable on white but fails contrast on dark backgrounds like `--color-brand` (#0B1D3A). For dark-background contexts (cookie banners, overlays, dark navs), use `--color-text-inverse` or keep explicit light-colored values with a comment explaining the contrast requirement.
- **CSS token name collisions across naming conventions**: When a token system has both bare-prefix tokens (`--band-limited: #EA580C`) and namespaced tokens (`--color-band-limited: #6B7280`), the similar names resolve to completely different values. This happens when legacy 5-band tokens coexist with spec 3-band tokens. Always verify which token you're referencing by checking the actual resolved value in `tokens.css`, not the name pattern.
- **Negative-margin background extension must respect mobile padding**: Using `margin-left: calc(-1 * var(--space-xl))` to extend a section's background to container edges only works if the parent container has at least `--space-xl` padding. When the parent padding reduces at mobile breakpoints (e.g., `--space-base` at 640px), the negative margin exceeds the padding and causes horizontal overflow. Either: (1) reset the negative margin at the breakpoint, or (2) use a different technique (e.g., pseudo-element with `position: absolute`).
- **Grid `max-width` must include padding**: When a grid container has `max-width: Npx` and `padding: 0 Mpx`, the columns compete for `N - 2M` pixels, not `N`. A `max-width: 1024px` container with 24px padding holds only 976px for columns. If the columns sum to 1024px (720 + 24 gap + 280), they won't fit. Set `max-width` to `column-sum + 2 × padding`.
- **`font-weight: 700` with Inter at 400/500/600 causes synthesized bold**: Inter is loaded at three weights. Any CSS rule using `--font-weight-bold` (700) causes the browser to algorithmically embolden the 600 weight, producing heavier/less-crisp rendering than native 600. Audit all `font-weight-bold` references after a font migration and decide per-element whether 600 is acceptable.
- **Curated list config files live in `data/lists/`**: One JSON file per list, slug matches filename. Files prefixed with `_` (e.g., `_example.json`) are excluded from sitemap and slug discovery. Config is loaded at request time via `_load_list_config()` — no caching (editorial content, infrequent access).
- **`get_all_states()` is binary — use manifest fields for finer tiers**: `get_all_states()` classifies states as `supported` (any active source) vs `coming_soon`. Since all 8 states have active federal data, they all show as "supported." For consumer-facing tier displays (full eval vs health-only), check specific manifest fields like `STATE_EDUCATION` status directly via `COVERAGE_MANIFEST`. The route handler's `_classify_states()` demonstrates this pattern.
- **Supported-state lists must derive from `COVERAGE_MANIFEST`, not hardcoded sets**: The set of fully-supported states (`NY`, `NJ`, `CT`, `MI`) changes as expansion states graduate. Hardcoding `{"NY", "NJ", "CT", "MI"}` in route handlers or templates drifts silently when a new state gets `STATE_EDUCATION: "active"`. Derive from `COVERAGE_MANIFEST` via `_classify_states()` or equivalent — one truth, many consumers. The `vote_state()` route and the homepage state dropdown both hardcode the list and will need updating when CA/TX/FL/IL launch. The homepage coverage pills and "Expanding soon" text are also hardcoded in the template — acceptable for editorial content but flag for update when expansion states change.
- **`_prepare_snapshot_for_display()` is the canonical migration pipeline**: All snapshot deserialization paths (view, export JSON, export CSV, list pages, city pages) must use this helper. When adding new backfill/migration logic, add it to this function — not inline in route handlers. There are now 7 deserialization paths: `view_snapshot()`, `export_snapshot_json()`, `export_snapshot_csv()`, the compare route, `view_city()`, `widget_badge()` (badge SVG route), and `api_widget_data()` (widget data API).
- **Denormalized columns for aggregation queries**: When a user-facing page needs to aggregate snapshots by a field stored in `result_json` (e.g., city name from `demographics.place_name`), add a denormalized column on the `snapshots` table instead of using `json_extract()` at query time. Populate on save via a shared `_extract_*()` helper, backfill on startup for existing rows, and add a composite index. Three save paths must be updated: `save_snapshot()`, `save_snapshot_for_place()` INSERT, `save_snapshot_for_place()` UPDATE. The city/state_abbr columns (NES-352) established this pattern.
- **Bulk `get_snapshot()` in loops is an N+1 pattern**: When a route handler needs `result_json` from multiple snapshots (e.g., computing dimension averages across a city's snapshots), each `get_snapshot()` call opens a new DB connection and parses 100KB+ JSON. Acceptable at current scale (dozens of snapshots per city) but would need a bulk query (`get_snapshots_by_ids()`) or result caching if page load times grow. Cache reuse within the loop (e.g., `first_full_snap` for Census lookup) avoids the worst duplication.
- **UI Design Specification reference**: The canonical UI spec is at `docs/UI_Design_Specification_v1.1.md`. Reference it for token values, component patterns, layout architecture, spacing rules, and accessibility requirements. When the spec and the code disagree, the spec is authoritative for new work; existing code should be migrated toward the spec incrementally.

## Key Patterns

### CLI Entry Point (cli.py, NES-262)
- **Canonical CLI**: `python cli.py evaluate "123 Main St, White Plains, NY"`. Subcommand pattern via argparse — extensible for future commands.
- **JSON to stdout by default** (pipe to `jq`), `--pretty` for human-readable (`format_result()`), `--verbose` for stage timings to stderr.
- **Lazy import for Flask isolation**: `from app import result_to_dict` is inside `_cmd_evaluate()`, not at module top. This keeps `--help` instant and avoids Flask bootstrapping (which fails without `SECRET_KEY` in production-like environments). The `--pretty` path uses `format_result()` from `property_evaluator.py` which has no Flask dependency. Future cleanup: extract `result_to_dict()` and its helpers to `serialization.py`.
- **`on_stage_complete` callback**: `evaluate_property()` accepts an optional `on_stage_complete(stage_name, elapsed_seconds)` callback that fires after each stage (success or failure). The CLI uses it for `--verbose` output. Distinct from `on_stage` (fires *before* stage, used by worker for DB updates).
- **Old CLI deprecated**: `property_evaluator.py:main()` still works but has hand-rolled JSON serialization that drifts from `result_to_dict()`. Use `cli.py evaluate` for canonical output.
- **Makefile**: `make evaluate ADDR="..." ARGS="--verbose"`.

### Overflow Mode (overflow.py, NES-263)
- **Presentation-layer truncation**: `overflow(items, limit, *, label, label_fn, dump_path, dump_fn) → OverflowResult`. Truncates large lists for display with a summary footer and optional JSON file dump. Zero Flask/DB dependencies — pure stdlib utility.
- **`label_fn` replaces `label` entirely**: When `label_fn` is provided, its return value is used as the complete label text in the summary. The `label` parameter is ignored. They are mutually exclusive in effect — `label_fn` does not modify or extend `label`.
- **Dump is opt-in and truncation-gated**: File dump only occurs when BOTH `dump_path` is provided AND truncation happens. No dump on non-truncated results even if `dump_path` is set. Dump failure logs a warning and returns `dump_path=None` — never raises.
- **Not wired into any consumer yet**: Standalone module. Integration with CLI (NES-262), batch mode, and agent layer happens in those tickets. The pipeline's existing `[:5]`/`LIMIT 50` slices are data-fetching limits — this utility is for display truncation at the presentation boundary.

### Async Evaluation (job queue)
- **POST /** with address: creates a job in SQLite, returns `{job_id}` immediately (no client timeout).
- **GET /job/<job_id>**: returns `{status, current_stage?, snapshot_id?, error?}` for polling. status: `queued` | `running` | `done` | `failed`.
- **worker.py**: one background thread per gunicorn worker (started via `gunicorn_config.py` post_fork). Polls for queued jobs, claims atomically, runs `evaluate_property(listing, api_key, on_stage=...)`, updates `current_stage` in DB, then saves snapshot and marks job done or failed.
- **Worker thread watchdog** (`app.py`): `before_request` handler checks `ensure_worker_alive()` every 30 seconds via `time.monotonic()` throttle. Catches the failure mode where gunicorn keeps serving HTTP (accepting job submissions, returning `job_id`s) but the daemon worker thread has crashed — jobs accumulate in `queued` status with no processing and no error signal. The watchdog lazy-imports from `worker` to avoid circular imports and is wrapped in `try/except` so failures never block request serving. `ensure_worker_alive()` delegates to `start_worker()` which handles DB init, stale job requeue, and thread creation idempotently.
- **Dead worker thread is silent**: When the worker thread crashes, the gunicorn process continues serving routes normally. `POST /` accepts addresses and returns `job_id`, `GET /job/{id}` returns `{"status": "queued"}` forever. There is no error — the system appears functional but jobs never progress. This was discovered during the seed evaluation sprint: the server accepted 80+ jobs that sat in `queued` indefinitely. The watchdog was added to detect and recover from this.
- Frontend: form submit → POST → get job_id → poll GET /job/{id} every 2s → show stage text → on done redirect to /s/{snapshot_id}.

### API Clients
- GoogleMapsClient wraps all Maps API calls
- Always use `timeout=API_REQUEST_TIMEOUT`
- Handle quota errors gracefully
- **Non-physical place filtering** (NES-211): `places_nearby()` and `text_search()` centrally filter out permanently closed and online-only businesses before caching/returning results. All downstream consumers get clean data without their own filter logic. To add a new online-only service, add to `_ONLINE_BUSINESS_EXACT_NAMES` (preferred) or `_ONLINE_BUSINESS_NAME_PATTERNS` (for regex). Filtered places are attached to `TraceContext.filtered_places` for audit.
- **Auditable heuristic filters**: When filtering API results by name/pattern heuristics, return a reason string (not just bool) so filtered items can be traced and false positives audited. Pattern: `_non_physical_reason() → Optional[str]` with typed reasons like `"closed_permanently"`, `"online_exact:<name>"`, `"online_pattern:<regex>"`.
- **Google Maps `radius` parameter is a location bias, not a strict filter** (NES-250): `text_search()` and `places_nearby()` with a `radius` parameter bias results toward the location but do NOT guarantee results stay within the radius. A query for "city center" near Ann Arbor can return "Southfield City Centre" 35 miles away if Google ranks it highly. Always validate result proximity with `_distance_feet()` (haversine, zero API cost) before accepting. When searching for generic concepts ("downtown", "city center"), prefix the city name via `reverse_geocode_locality()` to improve relevance.
- **Supplemental Text Search for chain coverage** (NES-258): Google Places Nearby Search returns max 20 results ranked by "prominence," which can exclude well-known chains (Kroger, Meijer) in suburban areas. Add a `text_search("supermarket", lat, lng, radius_meters=8000)` call alongside the type-based `places_nearby` calls — Text Search uses relevance ranking, surfacing different results. Pattern: wrap in `try/except` with `logger.warning` (supplemental, never blocks scoring), then merge into the same `all_stores` list before `_dedupe_by_place_id()`. The same pattern applies to any dimension where the 20-result Nearby Search cap drops known venues. Both `text_search()` and `places_nearby()` have per-evaluation caches — initialize both in `GoogleMapsClient.__init__`, not lazily via `hasattr`.
- **Reuse `requests.Session` for connection pooling**: `OverpassHTTPClient` uses a persistent `self._session` (created once in `__init__`) instead of a fresh `requests.Session()` per request. This enables TCP/TLS connection reuse across queries within the same gunicorn worker process. `requests.Session` is thread-safe for concurrent usage. Set `trust_env = False` on the session to skip proxy env var lookups.
- **OSM amenity elements can be nodes OR ways**: Amenities like playgrounds and dog parks are frequently mapped as area polygons (`way`) in OSM, not just point nodes (`node`). When querying Overpass for amenity features, include both `node` and `way` element types. Parse amenity tags from all element types — restricting to `node` only will miss area-mapped features, which are common in suburban parks.
- **Census Geocoder geography layers for place resolution**: The Census Geocoder returns multiple geography layers in a single response. For place-level data, check in order: "Incorporated Places" (cities), "Census Designated Places" (CDPs — named unincorporated communities), "County Subdivisions" (townships/towns — common in MI, NJ, NY, CT). Michigan charter townships are MCDs, not Incorporated Places — without the County Subdivisions fallback, ~40% of suburban Metro Detroit addresses return no place data.
- **COUSUB FIPS codes are unique within a county, not within a state**: When caching or keying data by County Subdivision (COUSUB) FIPS code, always include the county FIPS in the key. Two townships in different counties of the same state can share the same COUSUB code. Cache key pattern: `county_subdivision:{state}{county}{cousub}`.
- **Census place name cleaning**: Census returns names with type suffixes ("Novi city", "Commerce charter township", "Levittown CDP"). Strip with regex, ordering longer patterns before shorter (`charter township` before `township`) to avoid partial matches.
- **ACS vintage pinning**: `_ACS_BASE` URL is pinned to a specific vintage year (e.g., `2022/acs/acs5`). When bumping, verify all queried variable codes still exist in the new schema — the Census Bureau occasionally renames or retires variables between vintages.
- **Venue cache write path** (NES-290): `places_nearby()` and `text_search()` upsert every returned venue into `venue_cache` in `spatial.db` as a write-only side effect. Category is inferred at the call site — `_PLACE_TYPE_TO_CATEGORY` dict for `places_nearby(place_type=...)`, `_infer_text_search_category(query)` for `text_search(query=...)`. When adding a new `places_nearby()` or `text_search()` call with a type/query not yet in the mapping, add it to `_PLACE_TYPE_TO_CATEGORY` or `_TEXT_QUERY_CATEGORY_RULES` in `spatial_data.py` in the same commit — otherwise the venue gets stored with a raw Google type string as category, making cache-hit queries unreliable.
- **Venue cache read path** (NES-291): Before every `places_nearby()` / `text_search()` Google API call, `GoogleMapsClient` checks `VenueCache.query_venues()` for fresh cached venues. Cache hit requires: (1) venues exist in the category within radius with `last_verified` within TTL, AND (2) `_is_coverage_sufficient()` confirms a prior API search covered this area and returned ≤ cached count (via `venue_search_areas` table). Empty-area edge case: `_has_prior_search()` returns `[]` (not `None`) when a prior search found zero results — prevents repeated API calls for genuinely sparse areas. `text_search()` skips cache for `"other"` category (city center queries). TTLs: `_VENUE_CACHE_TTL_DAYS` in `spatial_data.py` (30 days for businesses, 90 for public amenities). `CoverageTracker` in `property_evaluator.py` logs which categories came from cache vs API per evaluation → `evaluation_coverage` table in `nestcheck.db`. Cache reads populate the in-memory `_places_cache`/`_text_search_cache` on hit so subsequent identical calls within the same evaluation get the in-memory hit.
- **Venue cache variable scoping**: When using `cat = _PLACE_TYPE_TO_CATEGORY.get(...)` in a `try` block followed by code that references `cat` outside the `try`, move the `cat` assignment before the `try` — otherwise an exception before the assignment leaves `cat` undefined and causes `NameError`. This applies to both the write path (`upsert_venues` + `record_search_area`) and coverage tracker recording.
- **Walk time cache** (NES-292): `WalkTimeCache` in `spatial_data.py` caches Distance Matrix API results (walk/drive times + distances) in `walk_time_cache` table in `spatial.db`. Keyed by rounded origin coords (5 decimal places ~1.1m) + destination `place_id`. Supports partial hits: cached pairs skip the API while only uncached pairs are sent to Distance Matrix. TTLs: 180 days walk, 90 days drive (separate `walk_calculated_at`/`drive_calculated_at` timestamps per mode). Unreachable destinations stored as `-1` sentinel (distinguishes from NULL = "not yet queried"). `walking_times_batch()` and `driving_times_batch()` accept optional `place_ids` param for cache integration; single-pair `walking_time()`/`driving_time()` accept optional `place_id`. All call sites in `property_evaluator.py` AND `green_space.py` thread place_ids from their Google Places dicts. `CoverageTracker` tracks `walk_times_from_cache`/`walk_times_from_api` counts → `evaluation_coverage` table in `nestcheck.db`. On API exception, partial cache hits are preserved (uncached entries filled with 9999) rather than losing all results.
- **`walking_times_batch` call sites span multiple modules**: `walking_times_batch()` is called from both `property_evaluator.py` (7 sites) and `green_space.py` (1 site). When adding a new parameter to batch methods (like `place_ids`), grep ALL modules — not just the primary one. The `green_space.py` call site was missed during initial NES-292 implementation because the CLAUDE.md documentation said "all call sites in property_evaluator.py" without noting the cross-module call. Always run `Grep` for the method name across the entire project before declaring call sites complete.

### Venue Scoring Calibration (scoring_config.py + property_evaluator.py)
- **Venue eligibility thresholds**: `VENUE_MIN_REVIEWS` and `VENUE_MIN_RATING` dicts in `scoring_config.py` gate which venues enter scoring. Venues below thresholds are still returned for display — only excluded from headline selection and dimension score computation.
- **Search types ↔ eligibility types contract**: Every Google Places type accepted in the eligibility filter (e.g., `["cafe", "coffee_shop", "bakery"]` at line 4280) must have a corresponding `places_nearby()` query upstream. A type in the filter but missing from the search silently produces zero matches for that category — no error, just empty results. When adding a new accepted type, add the API query in both `score_third_place_access()` and `get_neighborhood_snapshot()`.
- **Excluded types lists must stay in sync across scoring and snapshot functions** (NES-279): `score_third_place_access()` and `get_neighborhood_snapshot()` both define `excluded_types` for coffee filtering. When adding new excluded types, update BOTH lists. Similarly, keyword-supplemented `places_nearby()` calls added to a scoring function must also be added to the corresponding `get_neighborhood_snapshot()` branch — the calls share cache keys, so the snapshot function gets them for free, but it must issue the call to populate the cache when it runs first.
- **Type-based eligibility filters belong before rating/review filters**: For dimensions like fitness where Google's `type` parameter alone doesn't prevent misclassified businesses (e.g., real estate offices tagged `gym`), add a required-types + excluded-types filter after dedup but before the rating/review threshold. This removes non-relevant venues from both scoring AND neighborhood display — unlike the rating/review filter which keeps sub-threshold venues for display. This is intentional: a misclassified business shouldn't appear in display at all, while a low-review gym should.
- **Quality ceiling pattern** (NES-222): Walk-time proximity scores are capped by a single `QualityCeilingConfig` ceiling computed from three additive bonus signals: (1) sub-type diversity via `_classify_coffee_sub_type` (bakery/cafe/coffee_shop), (2) social bucket diversity via `_SOCIAL_BUCKET_MAP` (coffee/bakery/bar/restaurant/other), and (3) median review depth. Formula: `base_ceiling + diversity_bonus + social_bucket_bonus + depth_bonus`, capped at 10. The old double-ceiling system (`THIRD_PLACE_CATEGORY_CEILINGS` + separate `_compute_quality_ceiling`) was consolidated into this single configurable system.
- **Suppressed dimensions** (`points=None`): When no venues pass eligibility, return `Tier2Score(points=None, suppressed_reason=...)` instead of `points=0`. Template shows "—", composite scoring excludes the dimension from aggregation (reduces effective max). This avoids penalizing addresses for data gaps.
- **Drive-time scoring for venue dimensions** (NES-259): When a dimension has `_*_DRIVE_KNOTS` in `scoring_config.py`, wire drive-time scoring into the scoring function following this pattern: (1) find the best facility by walk score first, (2) fetch one `maps.driving_time()` call for that facility only when `best_walk_time > WALK_DRIVE_BOTH_THRESHOLD` (cost discipline — single Distance Matrix call, not per-place), (3) score with `apply_piecewise(DRIVE_KNOTS, drive_time) × quality_multiplier`, (4) `best_score = max(walk_score, drive_score)` — drive can only help, never hurt, (5) include `drive_time_min` in neighborhood places dict for template display, (6) wrap the drive lookup in `try/except` with fallback to walk score. The `_result_sections.html` template already handles `drive_time_min` with tiered walk/drive display (drive-only > 40 min walk, drive-first 25–40, walk-first 20–25, walk-only ≤ 20). Currently wired for fitness; coffee and grocery have drive knots defined but not yet wired.
- **Private scoring constants across modules**: `_FITNESS_DRIVE_KNOTS` (underscore-prefixed) is imported into `property_evaluator.py`. This works but is fragile. When all three dimensions have drive-time wiring, refactor `DimensionConfig` to include an optional `drive_knots` field so they're accessed via `SCORING_MODEL.<dim>.drive_knots` instead of cross-module private imports.

### Empty State Copy Library (copy_library.py, NES-319)
- **Standalone content module**: `copy_library.py` provides what/why/so_what copy for every failure mode (F1–F6 + `input_missing`). Zero dependencies — pure stdlib. Follows the `overflow.py` standalone module pattern.
- **`CHECK_NAME_ALIASES` must stay in sync with evaluator check names**: When adding or renaming a `Tier1Check.name` or `Tier2Score.name` in `property_evaluator.py`, update `CHECK_NAME_ALIASES` in `copy_library.py` in the same commit. `get_copy()` resolves aliases first — a missing alias causes a silent `None` return (graceful degradation, but the empty state copy won't render). Same class of sync bug as `scoring_config ↔ property_evaluator` imports.
- **`COPY_LIBRARY` keys are snake_case, evaluator names are mixed**: Copy library uses clean keys (`flood_zone`, `power_lines`). The evaluator uses display names for legacy checks (`"Flood zone"`, `"Road Noise"`) and snake_case for Phase 1B spatial checks (`"hifld_power_lines"`). The alias dict bridges this. When adding a new check, if the evaluator name is already snake_case and matches the copy library key, no alias is needed (passthrough).
- **`EVALUATION_FAILURE_COPY` is standalone**: F6 has no check context — it's module-level, not nested in `COPY_LIBRARY`. The dict has 42 entries; total including F6 is 43.
- **`{vintage_year}` placeholder in ejscreen F2**: Literal string in the `why` field. NES-264 wiring will handle interpolation from ingestion metadata at render time. Don't use f-string or `.format()` at the copy library level.

### Narrative Insights (NES-191+)
- Insight generators are pure functions: `dict → str | None`. Keep them side-effect-free for testability.
- Place insight logic in dedicated modules (not `app.py`) — `app.py` is for routes/views.
- **Dict field-name contracts**: Insight functions that consume dicts from `present_checks()` must use the field names that function actually produces (`"name"`, `"result_type"`, `"category"`, etc.) — not alternative names like `"display_name"` or `"check_id"`. When adding a new consumer of presented-check dicts, verify field names against `present_checks()` output, not against `_PROXIMITY_LABELS` keys or other lookup tables. Always add a `or "this hazard"` (or similar) terminal fallback when interpolating dict values into user-facing sentences to prevent sentence fragments from malformed input.
- Avoid duplicating utility helpers across modules (e.g., Oxford-comma join) — use a shared module.
- **`_PROXIMITY_LABELS` must use "a/an + singular noun" pattern**: Every entry is interpolated into sentences like `"This address is close to {label}"` and joined with `_join_and()`. Plural or article-free entries (e.g., `"high-voltage power lines"` instead of `"a high-voltage power line"`) produce awkward grammar when combined with other labels that do have articles.
- **Test assertions on prose should be semantic, not literal**: Insight generator tests that assert on exact prose fragments (`"driving"`, `"on the other hand"`, `"no parks"`) break during content design passes. Prefer assertions on structural properties (entity names present, `car_dependent` flag, key semantic words like `"car"`) over verbatim copy. When a content pass updates N prose strings, expect ~N test assertion updates — budget for it.
- **HTML-escape external data in `| safe` narrative generators**: Functions that return HTML strings rendered with Jinja's `| safe` filter (e.g., `generate_report_narrative`, any future insight generator returning HTML) must escape all external data (place names from Google Places API, user input, check headlines) via `markupsafe.escape()` before interpolation. Without this, a malicious place name like `<script>alert(1)</script>` executes in the browser. Pattern: `str(_html_escape(name))` at the point of extraction, not at interpolation.
- **`_join_labels()` requires a non-empty list**: The function has no internal guard for empty lists — `_join_labels([])` crashes with `IndexError`. When the input list is built from a filter that can reduce to zero elements (e.g., `[i for i in items if i]`), always guard the call: `if filtered: _join_labels(filtered)`. This is easy to miss when the upstream list is truthy but contains only falsy elements like empty strings.

### Check Display Metadata (app.py)
- Each Tier1Check `name` in `property_evaluator.py` needs entries in app.py: `_SAFETY_CHECK_NAMES`, `_CLEAR_HEADLINES`, `_ISSUE_HEADLINES`, `_WARNING_HEADLINES`, `_HEALTH_CONTEXT`.
- Only add headline entries for result states the check can actually produce. `_build_health_context()` returns `None` silently for missing keys.
- **Naming split**: Legacy checks use display names (`"Power lines"`, `"Gas station"`). Phase 1B spatial checks use `snake_case` (`"ust_proximity"`, `"hifld_power_lines"`). Match the convention of the source check function.
- New checks NOT in `_CHECK_SOURCE_GROUP` render individually (not collapsed).
- **Hazard tier classification** (NES-241): `_TIER_2_CHECKS` in `app.py` defines area-level indicators (EJScreen) that render in a compact collapsible section. All other safety checks default to Tier 1 (prominent cards). When adding a new safety check, it's Tier 1 by default — only add to `_TIER_2_CHECKS` if it's an area-level statistical indicator rather than a direct proximity hazard. The `hazard_tier` field is computed in `present_checks()` and backfilled in `view_snapshot()` for old snapshots. Template filters on `selectattr('hazard_tier', 'equalto', N)` — missing the field silently excludes the item from both tiers, so the backfill is critical.
- **Tier1Check serialization paths**: `Tier1Check` is serialized in three places — `result_to_dict()` (main snapshot), the compare route's serialization loop, and CSV export. When adding a new field to the dataclass, update ALL three. The compare path is easy to miss.
- **Nullable field propagation**: When widening a dataclass field from non-optional to `Optional` (e.g., `points: int → Optional[int]`), audit every consumer: arithmetic aggregation (`sum()`), string formatting (`f"{x} pts"`), template division (`score / max`), and all serialization paths. Each needs a null guard or filter. The `_scorable = [s for s in scores if s.points is not None]` pattern keeps aggregation clean.
- **Search radius vs threshold** (NES-203): Search radius can be wider than the warning threshold to report "Nearest: X (Y ft)" on PASS. Set `show_detail=True` on those PASS results. When nothing is found even within the expanded radius, the detail string should match the `_CLEAR_HEADLINES` wording (use the threshold distance, not the search radius).
- **Cascading distance filters must use explicit upper bounds** (NES-266): When a proximity check has tiered zones (FAIL ≤ 150m, WARNING 150–300m, PASS > 300m), every candidate filter must have BOTH a lower AND upper distance bound. A filter like `distance > FAIL_RADIUS and aadt >= THRESHOLD` without `and distance <= WARN_RADIUS` silently extends the warning zone to the full search radius (e.g., 600m instead of 300m). The unused threshold constant makes it look correct at a glance. Ground-truth validation caught this bug — unit tests didn't because the test distance (200m) was inside the intended range.
- **Presentation-layer suppression** (NES-196): To hide checks from display without changing evaluation or storage, filter in the route handler (e.g., `view_snapshot()`), not in `present_checks()` or `result_to_dict()`. Use `result = {**result, ...}` to shallow-copy the deserialized snapshot dict before modifying — never mutate it in-place, as a future caching layer would silently corrupt shared state. Suppressed metadata (e.g., count) should travel inside `result` dict, not as a separate template var, so Jinja includes and macros can access it without relying on inherited scope. Apply suppression to ALL routes that render the same data (snapshot, compare-health, etc.) to avoid inconsistent UX.
- **Confidence score caps** (NES-sparse-data): Dimension scores from data-confidence-aware scorers must be passed through `_apply_confidence_cap(score, confidence)` before building the `Tier2Score`. `estimated` caps at 8/10, `verified` uncapped, `not_scored` excluded from composite entirely. Apply consistently to every dimension that classifies confidence — omitting it lets sparse data produce artificially high ratings.
- **`apply_piecewise` returns float → round to int for `Tier2Score.points`**: `apply_piecewise()` and `apply_quality_multiplier()` return floats, but `Tier2Score.points` is `Optional[int]`. Every scoring path must round at the `Tier2Score` boundary: `points = int(max(cfg.floor, capped_score) + 0.5)`. The `score_road_noise` function established this pattern; all other dimensions using `apply_piecewise` must follow it. Omitting the round silently stores floats in an int field, causing "7.6/10" display artifacts and type-contract violations in serialization paths.
- **Snapshot migration functions must run on shallow copies**: Every route that deserializes a snapshot and runs migration functions (`_migrate_dimension_names`, `_migrate_confidence_tiers`) must shallow-copy first: `result = {**snapshot["result"]}`. There are currently 4 deserialization paths: `view_snapshot()`, `export_snapshot_json()`, `export_snapshot_csv()`, and the compare route. When adding a new migration function, wire it into ALL 4 paths — missing one causes silent data inconsistencies. The `view_snapshot` route copies via `{**result, ...}` during suppression; the others need explicit `{**snapshot["result"]}`.
- **Shallow copy only covers top-level keys**: `{**result}` copies the top-level dict, but nested structures (lists of dicts like `dimension_summaries`, nested dicts like `green_escape`) remain shared references to the stored snapshot. When backfilling keys on nested dicts, either shallow-copy the nested structure first (`dict(_ge)`, `[{**dim} for dim in dims]`) and reassign, or use a non-mutating pattern like `{**dim, "new_key": value}` in a comprehension. Mutating nested dicts in-place silently corrupts the stored snapshot under a future caching layer.
- **Shared helpers for serialization + backfill**: When `result_to_dict()` and `view_snapshot()` backfill need identical classification logic (e.g., `_dim_band()`), extract to a module-level function. Defining the helper inside `result_to_dict()` forces the backfill to duplicate the logic inline, which diverges silently when thresholds change. Pattern: module-level `_DIM_BANDS` tuple + `_dim_band()` function, called from both the serialization list comprehension and the backfill loop.
- **Backfill ordering and safe template defaults**: Functions that compute derived fields from migrated data (e.g., `_compute_show_numeric_score` consuming `data_confidence`) must run *after* the relevant migration (`_migrate_confidence_tiers`). Do not add legacy tier names to the derived function's allowlist as a substitute for correct ordering — that masks wiring bugs silently. Template fallbacks for backfilled display flags should use the *restrictive* value (`false` / hide), so a missing backfill surfaces visibly rather than silently showing incorrect data.
- **EJScreen cross-reference annotations** (NES-316): `_EJSCREEN_CROSS_REFS` in `app.py` maps address-level check names to EJScreen indicator fields. Annotation injection in `_prepare_snapshot_for_display()` must run AFTER the NES-241 hazard_tier backfill (which rebuilds dicts via `{**pc, ...}`). Adding a new cross-reference pair: add one dict to `_EJSCREEN_CROSS_REFS` with `address_checks` (list of check names), `ejscreen_field`, `threshold` (percentile), and `template` (under 120 chars per UI spec Section 4.11). Annotations only fire on `result_type == "CLEAR"` checks.
- **Composite scoring exclusion for not_scored**: Dimensions with `data_confidence == CONFIDENCE_NOT_SCORED` are excluded from both the numerator and denominator of composite scoring. This prevents both inflation (invented scores pulling average up) and deflation (zero scores pulling average down). The `_scorable` filter in `evaluate_property()` handles this.
- **Dimension name renaming** (NES-210): When renaming a `Tier2Score.name`, update ALL construction sites in the scoring function, update `TIER2_NAME_TO_DIMENSION` (keep old name as backward-compat alias), and add the old→new mapping to `_LEGACY_DIMENSION_NAMES` + `_migrate_dimension_names()` in app.py. The migration must run in ALL snapshot deserialization paths: `view_snapshot()`, `export_snapshot_json()`, `export_snapshot_csv()`, and the compare route. In `view_snapshot()`, run migration AFTER the `{**result, ...}` shallow copy to avoid mutating stored snapshot dicts.
- **Pre-computed sidebar variables**: `_result_sections.html` pre-computes `show_score`, `band_class`, `band_label`, `safety_checks`, `clear_count`, `warning_count`, `issue_count`, `safety_concerns` at template top for reuse in both main column and sidebar. When adding new sidebar data, compute it in the top block — don't duplicate Jinja filter chains inside both column and sidebar sections.
- **Composite scoring is equal-weight**: All scorable dimensions contribute equally to the 0-100 composite score (`tier2_total / tier2_max * 100`). The persona/weighted scoring infrastructure (NES-133) was removed. `TIER2_NAME_TO_DIMENSION` is retained solely for the compare route's dimension ordering.
- **Jinja2 JSON-LD trailing commas from filtered loops**: When a JSON-LD `<script>` block has a comma before a `{% for %}` loop with a filter (`if dim.points is not none`), the comma is emitted even when the filter produces zero items — resulting in invalid JSON that Google silently ignores. Pre-filter into a `{% set %}` variable (`selectattr('points') | list`) and guard the comma + loop on that variable being truthy. Invalid JSON-LD means the entire structured data block is silently dropped with no error in the page.
- **JSON-LD BreadcrumbList requires absolute URLs**: Google silently ignores `BreadcrumbList` items with relative URLs (e.g., `"item": "/state/ny"`). Always prepend `request.host_url.rstrip('/')` in the template: `"item": "{{ request.host_url.rstrip('/') }}{{ crumb.url }}"`. The `city.html` template established this pattern at line 24; `snapshot.html` was fixed in NES-344 to match. When adding breadcrumbs to new templates, always use the absolute URL pattern.
- **Cross-`try`-block variable scoping**: When a later `try` block references a variable assigned inside a preceding `try` block (e.g., `city_list` assigned in city pages block, consumed in state pages block), the variable is undefined if the first block throws before the assignment. Always initialize the shared variable before the first `try` (e.g., `city_list = []`) so the second block sees a safe default instead of `NameError`. Both blocks' `except` handlers catch the error, but the `NameError` message is misleading — it says the variable doesn't exist, not that the upstream data fetch failed.
- **OG image generation in worker must never block job completion**: The `generate_og_image()` call comes AFTER `complete_job()` and is wrapped in a broad `try/except`. If Pillow or fonts are unavailable, the evaluation still completes — the OG image just falls back to the static default via the serve route's redirect. Same pattern as email sending: post-completion side effects that degrade gracefully.

### Schema Migrations (models.py `init_db()`)
- **`CREATE TABLE IF NOT EXISTS` never updates existing tables**: If the production DB already has a table, the `CREATE TABLE` statement is a no-op — even when new columns have been added to the code's schema. Every column that was added to a `CREATE TABLE IF NOT EXISTS` block after the table was first created MUST have a corresponding `ALTER TABLE ADD COLUMN` migration in `init_db()`. Without this, `CREATE INDEX` on the missing column crashes with `OperationalError: no such column` and kills all gunicorn workers on boot. This caused a 2.5-hour outage (7 consecutive failed deployments).
- **Pattern for safe column + index addition**: (1) Fetch `PRAGMA table_info(table)` into a set, (2) `ALTER TABLE ADD COLUMN` if missing, (3) `CREATE INDEX IF NOT EXISTS`. The `job_cols` block for `evaluation_jobs` is the reference pattern. Never add a bare `CREATE INDEX` on a migration-added column without the `ALTER TABLE` guard above it.
- **Consolidate PRAGMA calls per table**: Each table's `PRAGMA table_info()` should be called once, stored in a variable, and reused for all column checks on that table. Don't scatter multiple PRAGMA calls for the same table across the function.
- **Railway healthcheck is a liveness probe, not a readiness probe**: `/healthz` must always return 200 once Flask is accepting connections. Returning 503 for missing env vars causes deployment failures. Report config status in the response body for monitoring, but never block deployment with a non-200 HTTP status.
- **When diagnosing deployment failures, read Deploy Logs first**: Build Logs show compilation; Deploy Logs show actual startup errors (`Worker failed to boot`, `OperationalError`, import crashes). The deploy logs contain the stack trace that identifies the root cause. Healthcheck timeouts are a symptom, not the cause.

### SQLite Concurrency (models.py)
- **`nestcheck.db` path does NOT auto-detect Railway volume**: `models.py` uses `os.environ.get("NESTCHECK_DB_PATH", "nestcheck.db")` — defaulting to the ephemeral working directory. In contrast, `spatial_data.py` auto-detects via `RAILWAY_VOLUME_MOUNT_PATH`. If `NESTCHECK_DB_PATH` is not set in Railway's dashboard (pointing to `/app/data/nestcheck.db`), every deploy wipes the operational database (jobs, snapshots, payments). This asymmetry caused a "Job not found" failure when a deploy swapped the container mid-evaluation — the new container had a fresh DB. Verify `NESTCHECK_DB_PATH=/app/data/nestcheck.db` is set in the Railway dashboard. Future fix: add `RAILWAY_VOLUME_MOUNT_PATH` auto-detection to `models.py` matching the `spatial_data.py` pattern.
- `_get_db()` sets `PRAGMA busy_timeout=30000` (30s) for write contention under concurrent gunicorn workers + evaluation threads.
- Job-critical write functions (`create_job`) retry up to 3× on `OperationalError` with "locked"/"busy" in the message, with progressive backoff. Read functions (`get_job`) retry once.
- Pattern for retry-safe DB functions: open connection inside `try`, use `finally: conn.close()`, catch `sqlite3.OperationalError` outside the connection block.
- `sqlite3.connect(timeout=N)` and `PRAGMA busy_timeout` both set the busy timeout — use only the PRAGMA (more explicit, avoids redundancy).

### Ground-Truth Calibration (scripts/generate_ground_truth_*.py + scripts/validate_ground_truth_*.py)
- **Generator/validator pairs**: Each spatial check has a generator (creates synthetic test points from real spatial.db data) and a validator (runs the actual check function and compares results). UST: point geometry, single threshold axis (distance). HPMS: linestring geometry, dual threshold axes (distance AND AADT). FEMA: polygon containment (inside/outside, not distance-based).
- **Polygon containment ground truth** (FEMA, SEMS): Unlike distance-based checks, polygon containment tests use `ST_PointOnSurface()` (preferred) or centroids for "inside" points and bbox-diagonal offset for "outside" points. Key gotcha: irregular/concave polygons can have centroids outside the polygon boundary — `ST_PointOnSurface()` guarantees an interior point, but always verify via `ST_Contains` and adjust the expected result. Similarly, "outside" points may land in neighboring polygons (flood zones and Superfund sites can be dense) — verify against ALL polygons in the table, not just the source one, with a retry loop (up to 5 bearings). The `_point_in_any_polygon()` query is the containment analogue of the `_nearest_facility_distance()` nearest-neighbor check used in distance-based generators. **Metadata-filtered containment** (SEMS/NES-271): When only a subset of polygons triggers FAIL (e.g., NPL status F/P for Superfund), add a third test category: points inside non-qualifying polygons that should still PASS. This validates the metadata filter, not just the geometry. Also verify non-qualifying inside-points don't overlap with qualifying polygons (skip with warning if they do).
- **Nearest-neighbor adjustment**: After generating a test point at a controlled distance from one facility, query the spatial index for the ACTUAL nearest facility. If a different facility is closer, adjust the expected result. Without this, dense areas produce false mismatches.
- **Threshold import pattern**: Try importing constants from `property_evaluator.py`; fall back to hardcoded values if the import fails (Flask side-effects). Print which path was taken. The hardcoded values MUST stay in sync — add a comment pointing to the canonical source: `scoring_config.py Tier1Thresholds.<field_name>` (NES-265 centralized all Tier 1 thresholds there). The module-level constants in `property_evaluator.py` (e.g., `HIGH_TRAFFIC_AADT_THRESHOLD`) are re-exports from `_T1 = SCORING_MODEL.tier1` — they exist for backward compatibility with ground truth scripts.
- **LINESTRING perpendicular offset**: For linestring data (HPMS), generate test points perpendicular to the segment bearing at the centroid. Use `ST_StartPoint`/`ST_EndPoint` for bearing, `ST_Centroid` for midpoint, then offset at bearing + 90°. The actual `ST_Distance` to the line may differ slightly from the offset distance because the test point's nearest approach may not be the centroid.
- **Dual-axis test design** (HPMS): Partition segments into pools by the non-distance axis (AADT above/below threshold). HIGH_TRAFFIC pool gets 3 distance variants (CLOSE/MIDDLE/FAR). LOW_TRAFFIC pool gets 1 variant (CLOSE only) to test that the non-distance condition prevents triggering.
- **Aggregate validation entrypoint**: `scripts/validate_all_ground_truth.py` discovers all `validate_ground_truth_*.py` scripts, matches each to `data/ground_truth/<dimension>.json`, runs them as subprocesses, and prints an aggregate summary. Adding a new dimension requires only a generator script, validator script, and ground-truth file — no changes to the aggregate runner. The runner parses `Matches:` / `Mismatches:` lines from validator stdout, so individual validators must preserve that output format.
- **Ground truth file location**: Committed ground-truth files live in `data/ground_truth/<dimension>.json` (not at the project root). Generated with fixed seeds (`--seed 42`) for determinism. Regenerate after changing thresholds, expanding spatial data, or widening geographic coverage.
- **Tier 2 ground truth is fundamentally different from Tier 1** (NES-272): Tier 1 generators create synthetic coordinates against spatial.db and test pass/fail checks. Tier 2 generators test scoring *functions* in isolation with synthetic scalar inputs (walk times, ratings, review counts) — no API calls, no spatial.db. Two sub-patterns exist: (a) **Single-curve dimensions** (coffee, grocery) test `apply_piecewise()` with types: knot boundary, interpolation, monotonicity, clamping, floor, ceiling, pipeline composition. Reference: `generate_ground_truth_coffee.py`. (b) **Multi-component additive dimensions** (parks) test each subscore function independently + composite totals, with types: walk_time, size_enriched, size_fallback, quality, nature_feel, composite, composite_cap, monotonicity, criteria. Reference: `generate_ground_truth_parks.py`. When the scored dimension has multiple subscore functions (not a single piecewise curve), use pattern (b) — test each subscore at its own thresholds, then test the composite sum and cap.
- **Pipeline composition tests guard against double-application bugs**: Sub-function tests (ceiling, floor, piecewise) verify individual operations. Pipeline composition tests replicate the *exact* order of operations from the scoring function (piecewise → ceiling → confidence cap → floor → round) and assert every intermediate value. This catches bugs where a cap is applied twice or in the wrong order — sub-function tests miss these because they test components, not composition. Always include pipeline composition tests when a scoring function has multiple sequential caps/transforms.
- **Tier 2 validator must also add to `_DIMENSION_LABELS`**: The aggregate runner auto-discovers `validate_ground_truth_*.py` scripts, but uses `_DIMENSION_LABELS` in `validate_all_ground_truth.py` for human-readable output. Add a label when adding a new Tier 2 dimension.
- **Tier 2 validators that pass `None` vs default dataclass must match the real caller**: When a scoring function accepts `Optional[SomeDataclass]`, the `None` and `SomeDataclass()` (default) code paths may diverge — e.g., `_classify_transit_confidence()` reads `frequency_bucket` via `getattr` on a non-None `TransitAccessResult`, producing different confidence than when `transit_access=None`. Validators must pass `None` when the JSON field is absent, not a synthetic default object. Similarly, generator scoring mirrors must replicate ALL guards from the real classifier (e.g., `has_hub = node_count >= 5`), not just the primary condition — omitting secondary guards produces correct results by coincidence on current test cases but breaks silently when new cases are added.
- **Legacy vs new string format mismatches in classifiers**: When a classifier checks string constants from multiple sources (e.g., `PrimaryTransitOption.frequency_class` = `"Very low frequency"` vs `TransitAccessResult.frequency_bucket` = `"Very low"`), the gate tuple must include BOTH formats. Ground truth validation is the fastest way to surface these mismatches — the bug is invisible in production because the classifier falls through to a nearby default that happens to be correct for common cases.
- **Band test fixtures should conform to the scoring model, not the other way around** (NES-273): When band tests fail because midpoint composites land on a different band than originally expected, update `expected_band` in the fixture to match the midpoint calculation — don't tweak synthetic inputs to force a target band. The midpoint-of-expected-ranges approach is more principled than hand-picked expected bands. Overfitting synthetic inputs to force a specific band is the wrong fix; it masks whether the scoring model produces sensible bands from plausible scores. When updating, verify the composite is within ±5 points of a band boundary to confirm it's a boundary-proximity issue, not a fundamental fixture error.
- **Extracted pure scoring functions enable testability** (NES-273): `compute_composite_score()`, `compute_park_score()`, `compute_transit_score()` are extracted as pure functions (no API calls) from `evaluate_property()`, `score_green_space()`, and `score_transit_access()` respectively. They delegate to the same internal subscore functions — no logic duplication. This pattern enables reference-address band tests without live API calls. When extracting a new pure scorer, keep it in the same module as the API-calling version so it stays in sync with internal helpers.

### Automation Scripts (scripts/daily_smoke_test.py, spatial_health_check.py, regression_baseline.py)
- **`result_to_dict()` vs `view_snapshot()` field contracts**: `result_to_dict()` produces the core evaluation dict (coordinates, tier1_checks, tier2_scores, final_score, neighborhood_places, etc.). Additional fields like `presented_checks`, `score_band`, `verdict`, `dimension_summaries`, `structured_summary`, and `walkability_summary` are added by `view_snapshot()` or other route handlers downstream. Scripts that validate result structure must validate against the contract of the function they actually call — `result_to_dict()` for evaluator-level validation, or the full snapshot dict for end-to-end validation. Validating `result_to_dict()` output against `view_snapshot()` keys produces false failures.
- **Railway volumes cannot be shared — use HTTP endpoints**: Each Railway service can only mount its own volume. The workaround for cron services that need production data is an authenticated HTTP endpoint on the web service. The spatial-health cron calls `GET /api/spatial-health` on the web service (authenticated via `SPATIAL_HEALTH_TOKEN`) instead of running its own ingestion. This pattern can be reused for future cron→web service communication.
- **Railway cron config is per-service via separate `.toml` files**: `railway.toml` configures the web service. Each cron service points to its own `railway-cron-*.toml` file via the "Railway Config File" dashboard setting. All three cron `.toml` files use `restartPolicyType = "NEVER"` since cron services must exit after completing their task.
- **Cross-script imports in `scripts/` need path setup**: Scripts in `scripts/` add `_PROJECT_ROOT` to `sys.path`, which enables imports from the project root (`property_evaluator`, `app`, etc.) but not from `scripts/` itself. `from regression_baseline import TEST_ADDRESSES` in `daily_smoke_test.py` works because Python adds the script's own directory to `sys.path` when run as `__main__`, but this breaks if the module is imported from elsewhere. Prefer inlining shared constants or extracting to a project-root module.
- **Importing from `scripts/` in app.py requires `scripts/__init__.py`**: `from scripts.spatial_health_check import check_health` needs `scripts/` to be a Python package. Without `__init__.py`, the import raises `ModuleNotFoundError` at runtime — easy to miss since it's a lazy import inside a route handler, not a startup import. The `__init__.py` exists (empty) specifically for this purpose. When adding new cross-boundary imports from `scripts/` into app code, verify the package file exists.
- **HTTP endpoints returning error status codes still carry valid data**: `_fetch_from_endpoint()` must handle 503 as valid JSON (unhealthy data), not as a network error. `urllib` raises `HTTPError` for non-2xx — catch it, read the body, and parse. Always guard `json.loads()` with `JSONDecodeError` handling when consuming HTTP responses, since proxies and load balancers can inject HTML error pages at any status code.

### Spatial Ingest (startup_ingest.py + scripts/ingest_*.py)
- `startup_ingest.py` calls `ingest()` directly with keyword args — scripts must accept kwargs, not just argparse.
- State filter format varies by dataset: 2-letter (`"NY"`) for EJScreen/TRI/NCES, full name (`"New York"`) for UST, FIPS code (`"36"`) for TIGER, postal abbreviation (`STABR='NY'`) for NCES. Always check the upstream API field.
- **UST ArcGIS API state name inconsistency** (NES-294, NES-304): The UST endpoint stores state names inconsistently — `"NewYork"` (no space) but `"New Jersey"` (with space). Both the WHERE clause and the returned data are affected: `State = 'New York'` returns 0 results but `State = 'NewYork'` returns 33K. `ingest_ust.py` includes both space and no-space variants in the IN clause to catch all records, and `_build_state_normalizer()` restores proper formatting at insert time. `coverage_config.py` uses `state_key_format: "full_name"` with `_STATE_FULL_NAME` lookup to match the normalized values. `startup_ingest.py` uses `_ust_missing_states()` to check per-state data availability (not just `_table_has_data()` which would skip ingestion if any state has rows). Other ingest scripts (TRI, EJScreen, HPMS) use 2-letter abbreviations and are unaffected. ParkServe (`ingest_parkserve.py`) also uses a `State` field from ArcGIS — when wiring it into startup_ingest, check for the same stripping issue.
- **Per-state missing detection is required for all multi-state tables** (NES-297): When adding a new state to `TARGET_STATES`, all state-filtered ingest paths must detect the new state's absence and trigger re-ingestion. Use `_missing_states_abbr()` for tables with 2-letter `$.state` in metadata, `_missing_states_fips()` for FIPS-prefix tables, or the generic `_missing_states()` with a custom SQL expression. HPMS is the only incremental source (re-ingests only missing states); all others re-ingest all states due to DROP TABLE. Follow-up: convert DROP TABLE scripts to incremental (per-state DELETE + INSERT) for efficient expansion.
- **Normalizing stored data requires updating downstream consumers in the same commit** (NES-294): When fixing the format of data stored in `metadata_json` (or any persistent store), audit all code that reads the old format. Workarounds may have been coded to match the broken data (e.g., `_STATE_NAME_NO_SPACE` in `coverage_config.py` was built to query `"NewYork"`). Fixing the write side without fixing the read side flips the mismatch direction — it passes tests but breaks silently in production. Grep for the old value pattern across the entire project before declaring the fix complete.
- ArcGIS bbox filtering: `geometry` (JSON envelope), `geometryType=esriGeometryEnvelope`, `inSR=4326`. Server-side filter. **Bbox is an AND filter with `where` clauses** — when both `states` and `bbox` are passed to an ArcGIS query, results must match BOTH. A tri-state bbox excludes non-Northeast states even if the state list includes them. State-filtered startup wrappers must NOT pass bbox.
- **Adding a new state checklist**: (1) Add to `TARGET_STATES` in `startup_ingest.py`, (2) create `data/<st>_district_performance.csv` + `scripts/ingest_<st>_performance.py` (clone NJ pattern), (3) add lazy-import wrapper + wire into `_STATE_EDUCATION_INGEST`, (4) add to `CSV_FILES` in `crosswalk_geoids.py` with state-specific suffix patterns in `_normalize()` (longer suffixes before shorter), (5) verify no bbox is passed from startup wrappers for state-filtered datasets.
- Wiring pattern: lazy-import wrapper → `_table_has_data()` → `_run_ingest()`. One failure never blocks others.
- **SpatiaLite DDL is NOT idempotent**: `AddGeometryColumn()` and `CreateSpatialIndex()` fail if the column/index already exists — unlike `CREATE TABLE IF NOT EXISTS`. When table creation can race (concurrent gunicorn workers on cold start), wrap these calls in individual `try/except` blocks. The `venue_cache` table in `VenueCache._ensure_table()` is the reference pattern. Regular indexes should use `CREATE INDEX IF NOT EXISTS`.
- **ArcGIS polygon ring orientation**: Clockwise = outer boundary, counter-clockwise = hole. Use shoelace formula (`_ring_is_clockwise()`) to classify rings when features may have disjoint parts (e.g., school districts with enclaves). The simpler `MULTIPOLYGON(((ring1),(ring2)))` pattern used by SEMS/FEMA only works when all rings belong to one contiguous polygon.
- **Two-table join pattern** (NES-206): Spatial polygon table (`facilities_school_districts`) for point-in-polygon → extract GEOID → join to a separate lookup table (`state_education_performance`) for enrichment data. Table contains multi-state data (NY, CT, NJ) with a `state` column; GEOID is unique across states. Different from single-table spatial checks.
- **Bundled CSV for data without stable APIs**: When upstream data is only available as Access DBs or manual downloads (e.g., NYSED), ship a curated CSV in `data/` and flag as `MANUAL REFRESH` in the `dataset_registry` notes. Include refresh cadence.
- **Point geometry ingest**: For point-location datasets (e.g., NCES schools), use `MakePoint(lon, lat, 4326)` instead of `GeomFromText('POINT(...)', 4326)`. Simpler and avoids WKT string building. Geometry comes as `{x: lon, y: lat}` from ArcGIS; always fall back to attribute fields (`LATCOD`/`LONCOD` for NCES 2022-23) if geometry is missing.
- **ArcGIS field name drift across dataset years**: Federal ArcGIS REST endpoints (NCES EDGE, TIGERweb) rename fields between releases without versioning. NCES 2022-23 changed `LEVEL` → `SCHOOL_LEVEL`, `LAT`/`LON` → `LATCOD`/`LONCOD`, and `SCH_TYPE_TEXT` became unqueryable in `outFields` (returns 400). When an ingest script fails with ArcGIS `{'code': 400, 'message': 'Failed to execute query.'}`, check field names with `outFields=*&resultRecordCount=1` first — don't assume the schema from a prior year still holds. Run `--discover` mode before full ingestion after any dataset year bump.
- **NCES suppression codes in numeric fields**: NCES uses negative integers (`-1`, `-2`) as suppression/missing-data indicators in otherwise-numeric fields like `TOTFRL` (Free/Reduced Lunch count). Always filter `< 0` values before arithmetic — they silently produce nonsensical results (e.g., `-0.3%` FRL rate) rather than raising errors.
- **Derived percentages from upstream counts**: When computing percentages from two upstream fields (e.g., TOTFRL/MEMBER for FRL%), cap at 100% (data quality issues can produce numerator > denominator) and filter out negative suppression codes (see above).
- **Template component duplication**: When the same UI component renders in multiple conditional branches (e.g., school card inside vs. outside school district section), extract into a Jinja macro in `_macros.html` immediately. Copy-paste diverges silently on the next edit.
- **Jinja macros with inline HTML strings need `| safe`**: When a macro builds an HTML string (e.g., an SVG icon) via `{% set _icon_svg = '<svg ...>' %}`, Flask's autoescape (enabled for `.html` templates) will escape `{{ _icon_svg }}` into literal `&lt;svg&gt;` text. Always use `{{ _icon_svg | safe }}` for internally-constructed HTML constants. This is safe because the string is hardcoded in the macro, not derived from user input. Never apply `| safe` to variables that contain external data.
- **Multi-state shared table idempotency**: When multiple scripts write to the same table partitioned by `state` column, never use `DROP TABLE` — use `CREATE TABLE IF NOT EXISTS` + `DELETE FROM ... WHERE state = 'X'` + INSERT. The DROP pattern destroys other states' data when run standalone.
- **Per-state data checking in startup_ingest**: When a table has rows from multiple independent ingestion sources (e.g., `state_education_performance` with NY/NJ/CT), use `_table_has_state_data(db_path, table, state)` to check each state independently. Using `_table_has_data()` (checks total rows) will skip ingestion for state B if state A's data already exists. Three datasets now use `_*_missing_states()` helpers: UST (full-name keys), EJScreen (abbrev keys, NES-302), HPMS (abbrev keys, NES-305). All query `DISTINCT json_extract(metadata_json, '$.state')`. **Drop-and-recreate vs incremental**: EJScreen's `create_facility_table()` drops the table, so ANY missing state triggers a full 8-state re-ingest. HPMS uses per-state DELETE+INSERT, so only missing states are passed via `states_override`. When adding a new per-state missing-states check, choose the pattern that matches the ingest script's idempotency model. Tech debt: the three helpers are near-identical — a generic `_missing_states(db_path, table, key_fn)` would eliminate ~60 lines of duplication.
- **Catch `IntegrityError` in CSV ingest loops**: When inserting from bundled CSVs into tables with UNIQUE constraints, always catch `sqlite3.IntegrityError` alongside `ValueError`/`KeyError`. A duplicate key in the CSV would otherwise crash the entire ingest, leaving partial data.
- **HPMS display-name resolution** (`_build_road_display_name`): HPMS `ROUTE_ID` is a coded segment identifier (e.g., "00000001__", "14000638__"), not a road name. Use `_looks_like_route_id()` to detect coded IDs — checks for `"__"` (double underscore, universal in HPMS route_ids across states) and pure-digit strings. Fallback chain: `route_name` (e.g., "SUNRISE HWY", "US 1") → `segment.name` (if not a route_id) → `"Route {route_number}"` → `route_id` as last resort. When building Tier1Check details for any result path (FAIL, WARNING, PASS-with-detail), always call `_build_road_display_name()` — don't inline the name resolution. Multiple code paths constructing the same detail string will diverge when the resolution logic changes. **NJ caveat**: NJ HPMS route_ids use `"00000001__"` format (8 zero-padded digits + `__`), which fails Python's `str.isdigit()`. Always use `_looks_like_route_id()` instead of `isdigit()` for route_id detection.
- **HPMS "Ref Rt" prefix cleaning**: NY HPMS encodes reference routes as `"Ref Rt 907M GRAND CNTRL PKWY"`. The ingest script strips the `"Ref Rt "` prefix and the numeric code (e.g., `"907M "`) to extract the human name. Other states may use different conventions.
- **Bundled CSV GEOIDs must match TIGER exactly**: The two-table join (`facilities_school_districts` → `state_education_performance`) uses string equality on GEOID. Fabricated or misformatted GEOIDs silently produce zero matches with no error. When adding or updating bundled education CSVs, verify GEOIDs against the TIGER WMS endpoint (`MapServer/14/query?where=GEOID='NNNNNNN'`). Use `scripts/crosswalk_geoids.py` as the reference crosswalk tool.
- **EJScreen endpoint removed by EPA (Feb 2025)**: The combined EJScreen ArcGIS service (`geopub.epa.gov/arcgis/rest/services/EJSCREEN/`) was taken offline as part of a policy action. The `ejscreen.epa.gov` domain no longer resolves. Fallback: PEDP (Public Environmental Data Partners) hosts a V2.32 mirror at `services2.arcgis.com/w4yiQqB14ZaAGzJq`. `ingest_ejscreen.py` probes combined endpoints first, falls back to PEDP. PEDP serves polygon geometry (not point centroids) — centroid is computed via vertex-average of the outer ring.
- **EJScreen PEDP `returnCountOnly` quirk** (NES-301): The PEDP mirror (`services2.arcgis.com`, `useStandardizedQueries: True`) returns HTTP 400 for `returnCountOnly=true` combined with `ST_ABBREV` WHERE clauses on non-cached states — but paginated record fetching (`resultRecordCount=2000`) works fine for all states. `STATE_NAME` filter works with `returnCountOnly`. The ingestion path (`_ingest_from_pedp`) paginates without pre-counting, so this doesn't block ingestion. If a future diagnostic or pre-check needs EJScreen row counts per state, use `STATE_NAME = 'Michigan'` (full name) instead of `ST_ABBREV = 'MI'`.
- **EJScreen V2.32 field changes**: `CANCER` (Air Toxics Cancer Risk) and `RESP` (Air Toxics Respiratory HI) were removed. `RSEI_AIR` (RSEI Air Toxics) replaced them. `CANCER_PCT` maps to `P_RSEI_AIR` in the PEDP fallback. `RESP_PCT` has no V2.32 equivalent — evaluator returns `None` for it (no health check triggers). If EPA restores the combined endpoint in the future, the old field names may return.
- **ArcGIS field discovery before choosing filter strategy**: Before deciding between bbox, state-field WHERE clause, or national ingest, query the endpoint's service definition (`FeatureServer/0?f=json`) to inspect available fields. Not all federal ArcGIS datasets have state-level attribute fields — HIFLD transmission lines have no STATE/STUSPS field (national ingest required), while FRA rail lines have `STATEAB` + `COUNTRY` (WHERE clause viable). The `--discover` CLI flag on ingest scripts fetches one sample record and prints attributes/geometry keys for quick inspection.
- **ArcGIS Online `outFields` with nonexistent field names returns 400**: Unlike `outFields=*`, requesting a specific field name that doesn't exist in the service causes a 400 "Cannot perform query" error with no indication of *which* field is the problem. When switching to a new data source version, always verify field names with `outFields=*&resultRecordCount=1` before hardcoding field lists.
- **ArcGIS Online orgs are not namespaced by purpose**: Service names like `EJScreen_Hazardous_Waste_Proximity` on `services2.arcgis.com` can belong to any organization (e.g., Utah DEQ, not EPA national). Always verify the service extent and sample records cover the target geography before writing ingestion code.
- **ArcGIS WHERE clauses fail silently on value mismatches** (NES-304): `State = 'New York'` returns 0 results (not an error) when the field stores `"NewYork"`. Always verify filter values with `returnCountOnly=true` for at least one expected-populated value before assuming a WHERE clause works. A 0-count response looks identical to "no data exists" vs "your filter value doesn't match the stored format." This is especially insidious for multi-state ingestion: 7 of 8 states succeed, one silently returns nothing, and the missing data isn't noticed until a user evaluates an address in that state.
- **FEMA NFHL MapServer returns 500 for large bboxes**: The FEMA flood zone endpoint (`hazards.fema.gov`) can't handle query envelopes larger than ~0.5° on a side. Chunk large-area ingestions into smaller tiles and handle per-tile failures gracefully. Some tiles (especially at state borders near lat 42°) consistently fail — this is a FEMA server limitation, not fixable on our side.
- **`TARGET_STATES` is the single source of truth for geographic scope** (NES-281): All state-filtered ingest calls derive their filter lists from `TARGET_STATES` dict in `startup_ingest.py`. Keys are 2-letter postal codes; values have `fips` and `full_name`. To add a new state for federal data: add one dict entry. Education performance data is separate — `_STATE_EDUCATION_INGEST` (at bottom of file) maps state codes to ingest wrapper functions; not all states in `TARGET_STATES` need education data immediately. HIFLD ingests nationally (no state field available). FRA uses `STATEAB` field driven by `TARGET_STATES` (NES-285). FEMA NFHL uses per-metro bboxes filtered by `TARGET_STATES` via `METRO_TO_STATES` mapping in `ingest_fema.py` (NES-286).
- **FEMA NFHL per-metro ingestion** (NES-286, NES-310): FEMA has no state attribute field, so filtering is geographic (bbox). `ingest_metros(target_states)` in `ingest_fema.py` selects metros where `METRO_TO_STATES[metro]` overlaps `target_states`, then subdivides each metro bbox into ≤0.5° grid cells via `_generate_grid_cells()` before querying. Coverage is metro-only — rural/exurban addresses outside defined metro bboxes will get UNKNOWN (no coverage) from `check_flood_zones()`. To add a new metro: add to `METRO_BBOXES` and `METRO_TO_STATES` in `ingest_fema.py`, and update the `--metro` argparse help string in the same commit. Per-chunk failures are logged and skipped — one bad tile never aborts the whole ingestion. **Version-based re-ingestion** (NES-310): `FEMA_INGEST_VERSION` constant at top of `ingest_fema.py` is bumped when bboxes change. `get_stored_fema_version()` reads `v=N` prefix from `dataset_registry.notes`. `startup_ingest.py` compares stored vs current version — mismatch triggers full re-ingest even when the table has data. `ingest_metros()` writes `v={VERSION}` prefix in registry notes after successful ingestion.
- **`json_extract()` on spatial.db is expensive for aggregate queries**: Per-state row counts via `json_extract(metadata_json, '$.state')` across tables with 200K+ rows take ~350ms per table. `verify_coverage()` (4 states × 10 tables) takes ~1.5s total. Never use `json_extract`-based counts in page-rendering paths — use static config for fast display, reserve DB queries for admin/diagnostic endpoints. The `/coverage` page learned this: switched from `verify_coverage()` (1,452ms) to manifest-only `get_source_coverage()` (0ms).
- **Don't declare planned-but-not-started sources in `_SOURCE_METADATA`**: A source in the metadata counts in the tier denominator for its dimension. A "planned" source with no ingestion work started drags the dimension to MINIMAL even when live API sources are fully functional. Only add a source to `_SOURCE_METADATA` (and `SOURCE_DISPLAY_LIST`, `_SOURCE_TO_REGISTRY_KEY`, per-state manifests) in the same commit as its ingest script. This avoids phantom coverage penalties.
- **Never run multiple `create_facility_table()` calls concurrently**: `create_facility_table()` does `DROP TABLE IF EXISTS` + `CREATE TABLE` + SpatiaLite DDL. Running two ingest scripts simultaneously causes one to DROP the other's table between CREATE and INSERT. Run ingestions sequentially from `startup_ingest.py` (which serializes them), never in parallel shell processes.
- **Coverage manifest auto-sync after ingestion (NES-309)**: `sync_manifest_from_db()` in `coverage_config.py` updates `COVERAGE_MANIFEST` in-place after `_check_and_ingest_all()` completes. Only syncs sources that have both a DB table AND a `state_filter` — national datasets (HIFLD: `state_filter=None`) and spatial-filter-required datasets (FEMA: `spatial_filter_required=True`) are skipped because total row counts can't determine per-state coverage. If a table is missing entirely (failed DROP TABLE re-ingestion), sync skips it rather than demoting — the data was there before and will return on next successful ingest. `_SYNC_VALID_TABLES` and `_SYNC_VALID_FILTERS` whitelists validate all f-string interpolated SQL, matching the `_VALID_FACILITY_TYPES` pattern in `spatial_data.py`.

### Comparison View (app.py + compare.html)
- **Structured differential pattern** (NES-207): Multi-snapshot comparison views compute differential data in a pure helper (`_build_comparison_data()`) called from the route handler, not in Jinja templates. The helper returns typed data structures (health grid, dimension rows, key differences) — the template only renders. This keeps business logic testable and templates simple.
- **Legacy/Phase 1B dedup in comparison grids**: When the same check exists in both legacy (`"Power lines"`) and Phase 1B spatial (`"hifld_power_lines"`) forms, use `_SPATIAL_SUPERSEDES` to skip legacy rows when the spatial version exists in any snapshot. Also deduplicate by display label (`seen_labels`) to prevent duplicate rows.
- **`score_ring` macro requires `report.css`**: The `score_ring` macro from `_macros.html` depends on `.band-exceptional`, `.band-strong`, etc. classes defined in `report.css`. Any template using the macro must load `report.css` even if it doesn't render full report sections.
- **Comparative verdict** (NES-218): `_build_comparative_verdict()` is a pure function that synthesizes headline + body from already-computed comparison data. It does NOT call APIs or touch the DB. When adding new comparison prose, add a branch to this function and a corresponding unit test — don't put text-generation logic in the template.
- **Branch-priority ordering for score spreads**: When multiple `if spread <= N` branches exist, check tighter thresholds first (e.g., `<= 3` before `<= 5`) to avoid the looser condition swallowing the tighter one. This is a common bug in cascading numeric thresholds — always order from most specific to least specific.

### Quality Ceiling (scoring_config.py + property_evaluator.py)
- **Quality ceiling pattern**: `QualityCeilingConfig` caps a dimension's max score based on three additive bonus signals: sub-type diversity, social bucket diversity, and review depth. Formula: `base(4) + sub_type_bonus + social_bucket_bonus + depth_bonus`, capped at 10. `_compute_quality_ceiling()` accepts `social_bucket_count` as a parameter — the caller computes bucket count via `_social_buckets_for_venue()` and passes it in. Applied *before* `_apply_confidence_cap()` so both constraints compose. To add a ceiling to another dimension: add `quality_ceiling=QualityCeilingConfig(...)` to its `DimensionConfig`, then call `_compute_quality_ceiling()` between the raw score and the confidence cap.
- **Threshold reachability**: Config thresholds must be reachable by the upstream classifier. Sub-type diversity: `_classify_coffee_sub_type()` returns at most 3 types, so `(4, bonus)` diversity thresholds are unreachable. Social bucket diversity: `_SOCIAL_BUCKET_MAP` maps to 5 buckets (coffee/bakery/bar/restaurant + other fallback), so thresholds up to 5 are reachable. Always verify the input domain before defining threshold tiers.
- **Pre-filtered inputs shift threshold floors**: `eligible_places` is already filtered to `reviews >= 15` and `rating >= 4.0` before reaching the ceiling. Depth thresholds near the filter floor (e.g., `(50, 0.5)`) will almost always trigger. Account for upstream filters when tuning thresholds.

### Display Thresholds (scoring_config.py)
- `WALK_DRIVE_BOTH_THRESHOLD` (20), `WALK_DRIVE_LEAD_DRIVE_THRESHOLD` (25), and `WALK_DRIVE_ONLY_THRESHOLD` (40) in `scoring_config.py` are the canonical walk/drive time display thresholds. Four display tiers: walk-only (≤20), walk-first-then-drive (20–25), drive-first-then-walk (25–40), drive-only (>40). All display logic (templates, drive-time fetching) should reference these, not hardcode magic numbers.
- **Scoring thresholds ≠ display thresholds**: `WALK_TIME_MARGINAL` (30 min, `green_space.py`) controls the walk-time *scoring curve*. `WALK_DRIVE_BOTH_THRESHOLD` (20 min) controls when drive times are *fetched and shown*. Don't conflate them — a constant can share a numeric value with a display threshold but serve a different purpose (scoring vs presentation).
- Transit drive-time fetch uses its own hardcoded threshold (`> 20` in `find_primary_transit`) — intentionally independent of the display constants. It follows a walk-primary/drive-secondary pattern where both are always shown when drive time exists.
- **Per-type transit search radii** (NES-317): `TRANSIT_SEARCH_RADII` in `property_evaluator.py` defines per-place-type radii for `find_primary_transit()`. `train_station` gets 16,000m (~10 mi) because commuter rail has drive-to catchments; `subway_station` and `light_rail_station` stay at 5,000m (walk-to). `evaluate_transit_access()` keeps its own 1,200m density radius — it measures walkable transit richness, not primary station discovery. When adding a new transit type to `find_primary_transit()`, add its radius to `TRANSIT_SEARCH_RADII` in the same commit.
- **Display-time derivation for cross-cutting widgets** (NES-249): When a UI component synthesises data from multiple dimensions (e.g., walkability summary from coffee/grocery/fitness/parks/transit walk times), derive it at display time in `view_snapshot()` rather than storing it in the snapshot. Pattern: pure function `_build_*(result) → dict | None`, called once before `render_template()`, assigned to `result["widget_key"]`. Benefits: works for old snapshots without migration, zero API cost, no `result_to_dict()` change, no export/compare wiring. Only add to `view_snapshot()` — not exports or compare routes.
- **`_best_walk_time()` filters by `rating is not None`**: Placeholder entries in `neighborhood_places` (e.g., "No full-service grocery nearby") have `rating: None` and `walk_time_min: 0`. Filter on `rating is not None` to exclude them. This doesn't apply to `green_escape.best_daily_park` or `urban_access.primary_transit` which are already validated by the evaluator.

### Payment (Stripe Checkout + Free Tier)
- **Payment state machine**: `pending` (checkout created) → `paid` (webhook confirmed or Stripe API verified) → `redeemed` (evaluation started). On eval failure: `redeemed` → `failed_reissued` (user can retry). Atomic CAS guards (`expected_status`) prevent TOCTOU races between webhook and return-from-Stripe.
- **Redeem-then-link gap**: `redeem_payment(token)` is called before `create_job()` (to gate creation), then `update_payment_job_id(token, job_id)` links them after. Server crash between these calls leaves a redeemed payment with no job_id — recoverable manually but not automatically.
- **Stripe return URL must URL-encode query params**: `success_url` carries `address`, `email`, `place_id` back through Stripe's redirect. These are user-supplied strings — use `urllib.parse.quote(value, safe='')` to prevent `&`, `=`, `#`, spaces from breaking URL parsing on return.
- **Cached snapshots bypass free tier recording**: The snapshot cache check (`is_snapshot_fresh`) can short-circuit before `record_free_tier_usage()`. This means repeated requests for a cached address don't consume the free credit — by design, since no new evaluation runs.
- **CSRFProtect instance must be stored**: `CSRFProtect(app)` returns the instance needed for `@csrf.exempt()`. Store in `_csrf` module variable. Webhook endpoint uses Stripe signature verification instead of CSRF.
- **Stripe graceful degradation**: Same pattern as OAuth — conditional import + module-level flag (`STRIPE_AVAILABLE`). When `stripe` package isn't installed or env vars are missing, checkout route returns 400 and app works in free-only mode.
- **`hash_email()` compute-once pattern**: When the POST `/` route uses email_hash for free tier check, job creation, and free tier recording, compute `email_h = hash_email(email) if email else None` once at the top of the payment gate block and reuse everywhere.

### Authentication (Flask-Login + Google OAuth)
- **`_FlaskUser` is not a DB model**: It wraps the dict from `get_user_by_id()` and implements Flask-Login's interface (`get_id()`, `is_authenticated`, etc.). Created fresh on every request via `user_loader`.
- **Open redirect prevention on OAuth callback**: `next_url` from session must be validated: only allow paths starting with `/` (not `//` which browsers treat as protocol-relative URLs). Pattern: `if not next_url or not next_url.startswith("/") or next_url.startswith("//"): next_url = "/my-reports"`
- **Snapshot claiming runs on every login**: `claim_snapshots_for_user()` matches by email on `WHERE user_id IS NULL AND email = ?`. Must run every login, not just `created=True`, because users may create evaluations with their email while logged out between sessions.
- **All snapshot save paths must include `user_id`**: Three INSERT paths exist: `save_snapshot()`, `save_snapshot_for_place()` (INSERT branch), and `save_snapshot_for_place()` (UPDATE branch — doesn't need `user_id` since it preserves existing). When adding a new column to snapshots, update all INSERT statements. Same pattern as documented for `Tier1Check` serialization paths.
- **Graceful degradation pattern**: Conditional import + module-level flag (`_oauth_enabled`). When OAuth env vars are missing: no Authlib import, no OAuth routes functional (redirect with flash), nav hides Sign In link via `{% if oauth_enabled %}`. App behavior is unchanged.
- **Never set `SESSION_COOKIE_DOMAIN` on a live deployment that previously used host-only cookies**: Flask's default (no `SESSION_COOKIE_DOMAIN`) creates host-only cookies — no `Domain` attribute, sent only to the exact host. Setting `SESSION_COOKIE_DOMAIN = ".example.com"` creates domain cookies (`Domain=.example.com`). If users' browsers still hold old host-only cookies, the browser sends BOTH on subsequent requests. Flask reads the first (the stale one) and ignores the new one — breaking session persistence. The `_redirect_to_canonical()` before_request hook makes domain pinning unnecessary: all traffic goes through one canonical host, so host-only cookies work correctly. If you ever need to switch cookie types on a live deployment, change `SESSION_COOKIE_NAME` simultaneously to avoid dual-cookie conflicts.

### Stripe Integration (Payments)
- **Payment status constants**: Use `PAYMENT_PENDING`, `PAYMENT_PAID`, `PAYMENT_REDEEMED`, `PAYMENT_FAILED_REISSUED` from `models.py` — never bare strings. Typos in status strings silently break state transitions with no error.
- **`_get_or_create_stripe_customer()` graceful degradation**: Customer creation failure falls back to `customer_email` pre-fill in the Stripe Checkout session, never blocks checkout. Pattern: try `Customer.create` → on failure, log warning and return `None` → caller checks and uses `customer_email` fallback.
- **Stripe webhook CSRF exemption**: Store `_csrf = CSRFProtect(app)` at module level and call `_csrf.exempt(view_func)` for webhook routes that use their own signature verification (Stripe-Signature header). Without storing the `CSRFProtect` instance, you can't exempt routes.
- **`REQUIRE_PAYMENT=True` requires `STRIPE_AVAILABLE=True`**: Pending payments cannot be verified without Stripe API access. The payment validation block must reject `pending` status when `STRIPE_AVAILABLE` is False — otherwise unverified payments proceed to job creation.
- **Stripe exception import path**: `stripe.error.SignatureVerificationError` was moved to `stripe.SignatureVerificationError` in stripe-python v6+. Use `except Exception as exc: if "SignatureVerification" in type(exc).__name__` for cross-version compatibility.
- **Free tier TOCTOU prevention**: `free_tier_usage.email_hash` must have a UNIQUE constraint. Use `INSERT OR IGNORE` + check `cursor.rowcount > 0` instead of SELECT-then-INSERT, which allows duplicate grants under concurrent requests.

This prevents health-severity colors from leaking into scoring or venue components.

The compact collapsible group (`health-tier2-section`) now contains only CLEAR items. This is a template-level split — the Python `hazard_tier` field still classifies checks as Tier 1 or 2.

### CI Pipeline (`.github/workflows/ci.yml`)
- **Two parallel jobs**: `scoring-tests` (merge gate, ~30s) runs `pytest tests/test_scoring_regression.py tests/test_scoring_config.py`. `ground-truth-validation` runs `scripts/validate_all_ground_truth.py` when `spatial.db` is available — skips with warning annotation when absent.
- **Makefile targets mirror CI**: `make test-scoring`, `make validate`, `make ci`. Always run `make test-scoring` locally before pushing scoring changes.
- **`spatial.db` is not in CI**: It's generated by `startup_ingest.py` from external APIs and is in `.gitignore`. Ground truth validation runs locally only until artifact provisioning is added.
- **conftest.py loads Flask app for all tests under `tests/`**: The `autouse=True` `_fresh_db` fixture and module-level `from app import app` import run even for pure-scoring tests. CI sets `SECRET_KEY` and `GOOGLE_MAPS_API_KEY` env vars to satisfy Flask startup. If scoring tests are ever too slow to load, move them to a top-level directory outside `tests/`.
- **`_fresh_db` fixture table cleanup list must include new tables**: When adding a new table to `init_db()`, add it to the `for table in (...)` loop in `tests/conftest.py:_fresh_db()`. Missing entries cause `OperationalError: no such table` in unrelated tests because the fixture runs `DELETE FROM` on every listed table before each test.
- **When adding new scoring test files to the CI gate**, update both `.github/workflows/ci.yml` (pytest command) and `Makefile` (`test-scoring` target) in the same commit.
- **Schema migration test** (NES-379): `tests/test_schema_migration.py` runs `init_db()` against a fixture DB with the oldest known production schema (no migration-added columns). When adding a NEW column to an existing `CREATE TABLE` in `init_db()`, do NOT update `_OLDEST_SCHEMA` in the test — the whole point is that it lacks those columns so the `ALTER TABLE` migration is exercised. When adding a NEW table to `init_db()`, DO copy it into `_OLDEST_SCHEMA`. When adding a new `ALTER TABLE` migration column, add it to the post-migration assertion block at the bottom of the test. `make test-schema` runs it standalone.

### Playwright Browser Tests (`tests/playwright/`)
- **Live Flask server via threading**: Playwright tests start a real Flask dev server in a daemon thread (`scope="session"`) because Playwright needs actual HTTP URLs — the Flask test client won't work. Server readiness is checked by polling `GET /robots.txt` with a timeout loop. Port is `5556` (hardcoded; should migrate to dynamic port via `socket.bind(('', 0))` if CI parallelism is added).
- **Fixture-to-snapshot pipeline**: Test fixtures are JSON files (`tests/fixtures/eval_result_*.json`) matching the `result_to_dict()` output shape. Each test saves a fixture to the DB via `save_snapshot()`, gets a snapshot URL, and Playwright navigates to it. `view_snapshot()` runs `_prepare_snapshot_for_display()` on the saved data, so the fixture must survive the full backfill pipeline.
- **`urban_access.primary_transit` is a single dict, not a list**: `_serialize_urban_access()` returns `primary_transit` as a flat dict (one transit option), not a list. Template code like `result.urban_access.primary_transit.walk_time_min` accesses it directly. `_build_walkability_summary()` calls `.get()` on it — passing a list crashes with `AttributeError: 'list' object has no attribute 'get'`. This was caught during the first Playwright test run.
- **Parent `_fresh_db` autouse fixture runs for Playwright tests too**: The `tests/conftest.py` `_fresh_db` fixture deletes all rows from core tables before each test. Playwright URL fixtures are function-scoped, so they save a fresh snapshot after each `_fresh_db` cleanup. This works because pytest runs setup sequentially: `_fresh_db` → URL fixture saves snapshot → test body runs. No race condition exists because the Flask server thread only reads during tests.
- **Fixture JSON must include all keys that `_result_sections.html` accesses without guards**: Jinja2's `is defined` and `|default()` silently hide missing fields — the template renders without error but the feature is inert. Playwright tests catch crashes (500 errors, `UndefinedError`) but NOT silent omissions. When building fixture JSON, trace every unguarded variable access in the template. The canary is `_prepare_snapshot_for_display()` — if it crashes on your fixture, a key is wrong.
- **Road noise renders inside `#health-safety` as a `.proximity-item`**: When asserting "all health checks are CLEAR," road noise with severity MODERATE renders as a `health_check_icon("warning")` inside a `.proximity-item` div. Scope health check assertions to `[id^='check-']` elements to exclude road noise, which has no `id` attribute.

### Railway Cron Services (`railway-cron-*.toml`)
- **`railway.toml` is single-service scoped**: It configures only the web service. Railway cannot define multiple services in one config file. Each cron service points to its own `railway-cron-<name>.toml` via the dashboard's "Config File Path" setting.
- **Railway volumes cannot be shared — HTTP endpoint workaround**: The web service owns `/app/data` with `spatial.db`. Cron services that need spatial data use an authenticated HTTP endpoint on the web service instead of their own volume. The spatial-health cron calls `GET /api/spatial-health` (Bearer token via `SPATIAL_HEALTH_TOKEN`) which checks the production volume directly. Env vars: `SPATIAL_HEALTH_TOKEN` (shared secret, set on both web and cron services), `NESTCHECK_WEB_URL` (cron-only, production URL).
- **Cron services must exit**: Railway skips subsequent cron executions if the previous one is still running. All automation scripts (`daily_smoke_test.py`, `spatial_health_check.py`, `regression_baseline.py`) are designed to exit 0 (success) or 1 (failure). Never add long-running loops or retry-forever patterns.
- **`restartPolicyType = "NEVER"` for cron**: Cron services should not auto-restart on failure — the next scheduled run handles retry. Only the web service uses `ON_FAILURE`.
- **Env var expansion in `startCommand`**: Shell variables like `$SMOKE_TEST_NOTIFY_EMAIL` in `startCommand` expand to empty string if unset, which passes `--email ""` to the script. Scripts should guard against empty-string email args (use `args.email or os.environ.get(...)` pattern).
- **Cross-script imports need `scripts/` on `sys.path`**: Scripts in `scripts/` that import from other scripts in the same directory (e.g., `from regression_baseline import TEST_ADDRESSES`) must add `_SCRIPT_DIR` to `sys.path`, not just `_PROJECT_ROOT`. The project root only covers top-level modules like `property_evaluator` and `app`. Pattern: `sys.path.insert(0, str(_PROJECT_ROOT))` then `sys.path.insert(0, str(_SCRIPT_DIR))`.
- **Three automation scripts and their Makefile targets**: `make smoke-test` (daily pipeline check), `make spatial-health` (weekly data integrity), `make regression` (monthly scoring stability). Email variants: `make smoke-test-email`, `make spatial-health-email`. Record baselines: `make spatial-baseline`, `make regression-update`.

### Safari Mobile / Viewport (iOS)
- `_base.html` sets `viewport-fit=cover` — required for `env(safe-area-inset-*)` to work. Do not remove.
- Never use bare `100vh` for layout heights; always add a `100dvh` override on the next line. Safari iOS `100vh` includes the URL bar's hidden space, which breaks scroll detection and makes the address bar stick in its collapsed state.
- Fixed-position elements pinned to `bottom: 0` must include `padding-bottom: calc(<normal> + env(safe-area-inset-bottom, 0px))` so they clear Safari's bottom bar and home indicator. Currently applied to: cookie banner (`base.css`), compare tray (`_compare_tray.html`).
- When adding new `position: fixed; bottom: 0` elements, follow the same `env()` pattern.

### Mobile Tab Bar + Section Navigation (NES-323)
- **`section-nav.js` uses a registration pattern for nav consumers** (NES-328): `SectionNav.register({ containerSelector, linkSelector, activeClass, onActivate })` adds a nav surface. The single `IntersectionObserver` calls all registered callbacks with the active section ID — it doesn't know which surfaces exist. Built-in consumers (desktop rail `.report-rail-nav`, mobile tab bar `.mobile-tab-bar`) are registered in the bootstrap IIFE at file bottom. To add a third nav surface, add one `SectionNav.register()` call — no changes to the observer. When adding a new report section, add the `data-section` link to BOTH `_report_rail.html` AND `_tab_bar.html`. Smooth-scroll-on-click uses `window.scrollTo` with a 52px top offset (`SCROLL_OFFSET` constant) so the target clears the sticky tab bar — never use `scrollIntoView()` for this because it ignores fixed/sticky headers.
- **Mobile tab order intentionally differs from desktop rail**: The PRD §6.2 specifies Verdict → Health → Dimensions on mobile (Health 2nd), while the desktop rail has Verdict → Dimensions → Health (Dimensions 2nd). Do not "fix" this — it's by design.
- **`_tab_bar.html` must be included in ALL templates that use `.report-layout`**: Currently `snapshot.html` and `index.html`. The tab bar is `display: none` at >= 1024px and sticky at `top: 0` below 1024px.
- **Never use `element.scrollIntoView()` to scroll within an `overflow-x: auto` container**: It can cause unwanted vertical page scroll on some browsers (Safari). Instead, use `container.scrollTo({ left: ..., behavior: "smooth" })` with manual offset calculation. The tab bar's active-tab centering uses this pattern.

### Embeddable Widget Templates (NES-349)
- **Standalone templates do NOT extend `_base.html`**: Widget templates (`widget_card.html`) are complete HTML documents with inline `<style>` blocks and zero external dependencies (no CSS, JS, or font imports). They use hardcoded hex colors mirroring `tokens.css` values, not `var()` references. System font stack only (`-apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif`) — no Inter, no Google Fonts.
- **`X-Frame-Options: ALLOWALL` is not a valid standard**: The only valid values are `DENY`, `SAMEORIGIN`, and `ALLOW-FROM origin`. Most browsers silently ignore `ALLOWALL`. For embeddable endpoints, use `Content-Security-Policy: frame-ancestors *` instead — it's the modern, standards-compliant equivalent. Omitting `X-Frame-Options` entirely also works (absence = allow framing) but CSP is more explicit.
- **`dict.get(key, default)` does NOT return the default when the key exists with a `None` value**: `result.get("final_score", 0)` returns `None` (not `0`) when `final_score` is stored as `None`. Use `result.get("final_score") or 0` to coalesce both missing keys and `None` values. This bit the widget card route where a `None` score would cause `TypeError` in `get_score_band()`.
- **Jinja2 autoescaping does not protect CSS context**: Template variables interpolated inside `<style>` blocks (e.g., `width: {{ width }}px`) are not escaped for CSS injection. Protection must come from the Python side — `request.args.get("w", 300, type=int)` coerces to int so non-int values return the default. When interpolating user-controlled values into `<style>` blocks, always validate/cast in the route handler.
- **Widget endpoints should NOT call `increment_view_count()`**: Iframe embeds on third-party sites would inflate snapshot view counts. Widget routes skip view tracking intentionally.
- **Badge route (`widget_badge`)**: `GET /widget/badge/<snapshot_id>.svg?style=banner|square`. Returns SVG with `Content-Type: image/svg+xml`. Missing or stale snapshots return a generic "Evaluate on NestCheck" fallback SVG (200, not 404) via `widget_badge_fallback.svg` — embeds should never show a broken image. Route maps `band["css_class"]` to hex color via `band_colors` dict (unlike card widget which maps in Jinja2). Invalid `style` param defaults to `banner`. SVG templates use `.svg` extension (not `.html`) — Jinja2 autoescaping is OFF for `.svg` files, so only interpolate server-controlled values. SVG uses `href` (not deprecated `xlink:href`). The SVG-internal link uses a relative URL — only functional when opened directly in browser, not when embedded via `<img>`. The embed snippet wraps the `<img>` in an `<a>` tag for clickability.
- **Widget data API (`api_widget_data`)**: `GET /api/v1/widget-data/<snapshot_id>`. Returns JSON with `score`, `band`, `address`, `health_summary`, `clear_count`, `concern_count`, `report_url`. 404 uses `jsonify()` (not `abort()`) so JSON consumers get parseable errors. `health_summary` string uses same pluralization as card widget template. `report_url` is absolute (uses `request.host_url`).
- **Embed modal**: Vanilla JS in `_result_sections.html` inside a `{% if not is_preview and snapshot_id %}` script block. Three tabs (Badge/Card/API). Badge tab fetches live SVG preview and updates snippet on style toggle. API tab fetches from widget-data endpoint on modal load. Modal CSS in `report.css` (`.embed-*` prefix). Hidden in print CSS. `copyEmbedSnippet()` is a global function (referenced from `onclick` attributes). Uses pre-computed `band_label` template variable (not `result.score_band.label`).

## Workflow Orchestration

### 1. Plan Mode Default
- Enter plan mode for ANY non-trivial task (3+ steps or architectural decisions)
- If something goes sideways, STOP and re-plan immediately — don't keep pushing
- Use plan mode for verification steps, not just building
- Write detailed specs upfront to reduce ambiguity

### 2. Subagent Strategy
- Use subagents liberally to keep main context window clean
- Offload research, exploration, and parallel analysis to subagents
- For complex problems, throw more compute at it via subagents
- One task per subagent for focused execution

### 3. Self-Improvement Loop
- After ANY correction from the user: update CLAUDE.md with the pattern
- Write rules that prevent the same mistake
- Ruthlessly iterate on these lessons until mistake rate drops

### 4. Verification Before Done
- Never mark a task complete without proving it works
- Diff behavior between main and your changes when relevant
- Ask yourself: "Would a staff engineer approve this?"
- Run tests, check logs, demonstrate correctness

### 5. Demand Elegance (Balanced)
- For non-trivial changes: pause and ask "is there a more elegant way?"
- If a fix feels hacky: "Knowing everything I know now, implement the elegant solution"
- Skip this for simple, obvious fixes — don't over-engineer

## Core Principles

- **Simplicity Bias**: Before implementing any solution, ask: "How can we make this simpler and dumber while still achieving the goal?" Prefer the dumbest thing that works. If a feature needs a config system, ask if a hardcoded value works first. If a function needs three parameters, ask if it can work with one. Complexity must be justified — simplicity is the default.
- **No Laziness**: Find root causes. No temporary fixes. Senior developer standards.
- **Minimal Impact**: Changes should only touch what's necessary. Avoid introducing bugs.

## When Unsure

- Ask clarifying questions before implementing
- Check existing patterns in the codebase first
- Prefer simple solutions over clever ones
