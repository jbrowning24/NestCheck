# NestCheck

Property evaluation tool for Westchester County rentals. Analyzes health, lifestyle, and budget criteria for Metro-North corridor properties.

## Tech Stack

- **Backend:** Python/Flask
- **Database:** SQLite with async evaluation queue
- **APIs:** Google Maps, Overpass (OpenStreetMap)
- **Frontend:** Jinja templates, vanilla JS
- **Hosting:** Railway

## Project Structure

```
NestCheck/
├── app.py              # Flask routes, API endpoints
├── models.py           # SQLite models, job queue
├── worker.py           # Background evaluation worker
├── property_evaluator.py  # Core evaluation logic, API clients
├── nc_trace.py         # Request tracing
├── templates/          # Jinja HTML templates
├── static/             # CSS, JS assets
└── issues/             # Local issue tracking (markdown)
```

## Claude Code Configuration

### Custom Commands & Hooks
- Custom slash commands in `.claude/commands/` must not shadow built-in commands (e.g., `/review`, `/compact`, `/init`). Check the built-in list before naming a new command. Shadowed built-ins become unreachable.
- PostToolUse hooks that reference `$CLAUDE_FILE_PATH` must guard with `test -n` and quote the variable. If unset, tools like `ruff format` will silently operate on the entire directory.
- Subagent definitions live in `.claude/agents/`. Each agent should have a single focused responsibility and explicit verification steps.
- SessionStart hooks that run `sudo apt install` will hang in environments requiring a password prompt. Always guard with `command -v <tool>` to skip installation when the tool already exists, and end with `|| true` so failures never block the session.

## Coding Standards

- Python: Follow existing patterns in property_evaluator.py
- Use type hints for function signatures
- Add docstrings for public functions
- No print() in production - use logging
- All API calls need timeout handling
- **Never use predictable defaults for secrets**: `os.environ.get("SECRET", "default-value")` is a security vulnerability if the default is guessable and the env var isn't set in production. For `SECRET_KEY`, `BUILDER_SECRET`, and similar auth/session secrets: use `os.environ.get("VAR")` (returns `None`) and guard all usage with `if VAR and ...`. For `SECRET_KEY` specifically, fail fast if it's the dev default and `RAILWAY_ENVIRONMENT` is set.
- **SQL table name interpolation must use whitelists**: f-string table names in SQLite (e.g., `f"SELECT ... FROM {table_name}"`) cannot be parameterized with `?`. Always validate against a whitelist before interpolation. In `spatial_data.py`, `_VALID_FACILITY_TYPES` is the canonical whitelist; `_validate_facility_type()` returns `None` for unknown types (graceful degradation in queries) or the validated table name. When adding a new spatial dataset, add its facility_type to `_VALID_FACILITY_TYPES` in the same commit as the ingest script.
- **Never store raw `str(e)` in user-visible DB fields**: Exception messages can contain file paths, SQL queries, API keys from error URLs, or internal stack details. Use `_sanitize_error(e)` in `worker.py` to classify exceptions into safe user-facing messages. Full details go to `logger.exception` only. The sanitizer matches on keyword presence (`"timeout"`, `"api"`, `"geocod"`, `"config"`) and returns one of five hardcoded strings.
- **Unbounded class-level caches need eviction**: Any `Dict` or `dict` used as a process-lifetime cache (shared across instances via class variables) must have a size bound. Use `OrderedDict` + `move_to_end()`/`popitem(last=False)` for LRU, or `functools.lru_cache` for function-level caching. `urban_access.py`'s `_cache_set()` is the reference pattern (1000-entry LRU with `threading.Lock`).
- Spatial metadata values may arrive as strings after JSON round-tripping — cast to `float()` before numeric formatting (`:,.0f`)
- **scoring_config ↔ property_evaluator import sync**: When adding new constants to `scoring_config.py` that are referenced in `property_evaluator.py`, update the `from scoring_config import (...)` block in the same commit. Missing imports surface as `NameError` only at evaluation runtime (inside try/except), not at startup — so they silently produce 0/10 scores with error text in reports instead of crashing visibly.
- When changing template element IDs, update `smoke_test.py` markers (`LANDING_REQUIRED_MARKERS`, `SNAPSHOT_REQUIRED_MARKERS`) in the same commit. Mismatches cause silent post-deploy smoke test failures.
- When removing HTML elements from templates, remove the corresponding CSS rules in the same commit. Orphaned selectors (e.g., `.snippet-assessment-score` after removing the score div) accumulate silently.
- When promoting a CSS pattern from a page-specific file to `base.css` (e.g., sticky footer), remove the now-redundant rules from page-specific files (e.g., `error-page.css`) in the same commit. Since page-specific files load after `base.css`, the duplicates are silently harmless but accumulate as confusing dead code.
- **Page-scoped CSS class naming**: Generic class names in page-specific stylesheets (e.g., `.section-label` in `pricing.css`) can collide with identically-named rules in globally-loaded stylesheets (e.g., `report.css`). If a page-specific class name is generic, prefix it with the page name (`.pricing-section-label`) to prevent silent style bleed if stylesheets are ever co-loaded.
- **External links always need `rel="noopener noreferrer"`**: `noreferrer` implies `noopener` and additionally suppresses the `Referer` header. Use both on every `target="_blank"` link — `rel="noopener"` alone is incomplete.
- **Inline SVG icons over HTML entities**: Use inline SVGs (e.g., `<svg width="12" height="12" viewBox="0 0 24 24" ...><polyline .../></svg>`) instead of HTML character entities (`&#9654;`, `&#9888;`, `&#9650;`) for icons. SVGs render crisply at all sizes, inherit `currentColor`, and animate smoothly via CSS `transform`. When replacing entities with SVGs inside an existing `.collapse-icon` span, switch the span from `display: inline-block` to `display: inline-flex` and add `.collapse-icon svg { display: block }` to eliminate baseline gaps. When replacing icons across the codebase, audit ALL templates in one pass — `_result_sections.html`, `compare.html`, `compare_health.html`, `_eval_snippet.html` — to avoid inconsistent rendering between pages.
- **Score display format consistency**: All `/10` scores in the report (dimension scores, park Daily Value, park subscores, green space list scores) should display as integers (`%.0f`). Google Places star ratings (`4.5 ★`) keep one decimal (`%.1f`) since that's their natural precision. Mixing formats (e.g., "7.6/10" next to "8/10") looks like a bug.
- **New visual sub-palette vs. token reuse**: When a page intentionally introduces a neutral/monochrome sub-palette (e.g., `#111111`, `#666666`) that diverges from the site's blue-tinted token system, use tokens for *structural* values (spacing, font weights, border-radius, transitions) but leave palette colors as explicit values rather than mapping them to semantically mismatched tokens. Forcing `#111111` into `--color-text-primary: #0F172A` (navy) or `#666666` into `--color-text-muted: #64748b` (cool blue-gray) produces subtle hue drift that undermines the design intent. Within the sub-palette, consolidate similar grays (e.g., `#555555` and `#8E8E8E` both → `#666666`) to avoid the "narrow band of grays" AI-slop signal.
- **No hover effects on non-interactive elements**: Cards, scorecards, and preview panels that are display-only (no `<a>`, no click handler) must not have `:hover` styles (translateY, shadow escalation, border-color changes). Hover lifts on static content is a top AI-slop pattern — it signals interactivity that doesn't exist and makes the page feel template-generated. Only add hover transitions to genuinely clickable elements.
- **Use `color` tokens for text hierarchy, not `opacity`**: `opacity` dims the entire element — text, borders, backgrounds, shadows, child elements — and doesn't compose predictably when multiple opacity rules overlap (see CSS opacity stacking entry below). For text de-emphasis, use explicit color tokens (`--color-text-muted`, `--color-text-faint`, `--color-text-secondary`). Reserve `opacity` for overlays, disabled states, icon badges, and elements where dimming *everything* is the actual intent.
- When relocating template sections to a different parent container, update surrounding copy (section headings, divider descriptions, context notes) to reflect the new contents. Stale copy (e.g., "Community and school data" after adding EJScreen) silently misleads users.
- Frontend `fetch()` calls that expect JSON must check `resp.ok` and content-type before calling `.json()`. Non-JSON error responses (CSRF 400, HTML 500) cause Safari-specific `TypeError` ("The string did not match the expected pattern") that hides the real error. Always guard: `if (!resp.ok) { /* handle non-JSON */ }` before `resp.json()`.
- Frontend polling loops must never give up on a single transient error (404, 5xx, network). Use retry counters with a cap before showing a failure. The job queue is eventually-consistent under load; a freshly-created job may not be visible to the poll endpoint for 1-2 cycles.
- Flask error handlers (`@app.errorhandler`) must return JSON when `_wants_json()` is true. Without this, JS clients get HTML error pages they can't parse. Add handlers for 400, 404, and 500 at minimum.
- **CSRF auto-retry pattern**: `csrfFetch(url, options)` in `index.html` wraps `fetch()` with CSRF token injection, `resp.ok`/content-type guarding, and single auto-retry on `error_code === 'csrf_expired'`. Use `csrfFetch` for all new POST endpoints instead of raw `fetch` + manual `X-CSRFToken` header. The server-side `@app.errorhandler(400)` adds `"error_code": "csrf_expired"` to JSON responses when the error message contains "csrf" — client matches on that structured code, not on substring search of the human-readable error text.
- **Google Maps async loading**: Maps API script tags must include `loading=async` URL parameter and `async` HTML attribute. Autocomplete init must use a named callback (`callback=initAutocomplete`) instead of an IIFE, since the `google` global isn't available at script parse time with async loading. When both a sync-check path and an async callback can init the same widget, guard with a flag to prevent double initialization.
- **Never use synthetic DOM events to trigger existing handlers**: `dispatchEvent(new Event('submit'))` creates a non-cancelable event that bypasses HTML5 validation and behaves differently from real user events. Instead, call the handler function directly (e.g., `submitEvaluation(addr, null, email)`) — it's clearer, more robust, and avoids subtle bugs with `preventDefault()` being a no-op on non-cancelable events.
- **Nested flex containers need their own `gap`**: CSS `gap` only applies between direct children of a flex container. When dynamically-generated items (e.g., pill buttons) are inside a wrapper `<div>` that is itself a flex child, the parent's `gap` spaces the wrapper from siblings — not the items inside it. Always add `display: flex; gap: Npx;` to the wrapper too, or append items directly to the flex container.
- **Cookie-based client state pattern**: `nestcheck_recent` cookie stores a JSON-encoded array of `{address, snapshot_id, place_id}` objects (max 5, 30-day expiry, `SameSite=Lax; Path=/`). `getRecentSearches()` handles graceful migration from the legacy string-array format (checks `typeof parsed[0] === 'string'`). Read with regex on `document.cookie`, write with `setCookie(name, value, days)`. Always wrap `JSON.parse` in try/catch and return `[]` on failure — cookies can be corrupted by browser extensions or manual edits. The `addRecentSearch` / `getRecentSearches` / `renderRecentSearches` / `updateRecentSearchSnapshot` functions in `index.html` are the reference implementation. Pill clicks with a `snapshot_id` check `/api/snapshots/<id>/fresh` before redirecting — stale or missing snapshots fall through to a new evaluation.
- **`get_snapshot()` loads full `result_json`**: The `get_snapshot()` function in `models.py` does `SELECT *` and parses `result_json` (potentially 100KB+). For lightweight checks (existence, freshness), this is wasteful. If a high-frequency endpoint only needs metadata columns (`evaluated_at`, `snapshot_id`), consider adding a `get_snapshot_metadata()` function that selects only non-blob columns. Currently `api_snapshot_fresh` uses `get_snapshot()` — acceptable for now but worth optimizing if the endpoint sees high traffic.
- **Template ↔ backend data contract verification**: Before adding Jinja2 template code that accesses a new field (e.g., `dim.band`, `result.neighborhood_summary`), verify the field is actually populated in the backend (`result_to_dict()` in `app.py` or the evaluator). Jinja2's `is defined` guards and `|default()` filters silently hide missing fields — the template renders without error but the feature is inert. Always grep the backend for the field name before writing template code that consumes it. This class of bug is invisible in smoke tests and only surfaces during visual QA.
- **CSS opacity stacking on dual-classed elements**: When adding an opacity-based modifier class (e.g., `.section-card--quiet { opacity: 0.85 }`) to elements that already have an opacity rule (e.g., `.place-card--muted { opacity: 0.75 }`), the later-declared rule wins in CSS cascade — it doesn't multiply. This can unintentionally *increase* opacity. Add a compound selector (`.place-card--muted.section-card--quiet { opacity: 0.75 }`) to preserve the original intent.
- **Remove `opacity` when migrating to semantic color tokens**: If a class previously had no explicit `color` (inheriting a dark parent color) and used `opacity: 0.7` to soften it, migrating to an already-faint color token (e.g., `--type-l5-color: var(--color-text-faint)`) while keeping the `opacity` produces double-faintness — 70% opacity of an already-light gray becomes near-invisible. When adding an explicit `color` that carries the intended faintness, remove the `opacity` rule.
- **CSS `var()` fallbacks are dead code when the token is defined**: `var(--color-bg, #ffffff)` looks like it defaults to white, but if `--color-bg` resolves to `#F8F9FB` (off-white), the fallback is never reached. Always verify the token's actual resolved value in `tokens.css`, not the fallback. Use `--color-bg-card` (`#FFFFFF`) for true white backgrounds on card surfaces, `--color-bg` / `--color-bg-page` (`#F8F9FB`) for page-level backgrounds. Match fallback hex values to the token's definition to avoid confusion.
- **Semi-transparent backgrounds depend on container color**: `rgba(255, 255, 255, 0.8)` is visible on a yellow background but invisible on white. When changing a container's background color, audit child elements for semi-transparent backgrounds that relied on contrast with the old color. Replace with opaque token values (e.g., `var(--color-border-light)`) that work regardless of the container.
- **Button classes on mixed elements**: `.share-btn` is used on both `<button>` and `<a>` elements. The base class includes `text-decoration: none` and `color: var(--color-text-inverse)` to ensure anchors look identical to buttons. When promoting dropdown menu items to direct buttons, always check if the target element type changes (`<button>` in menu → `<a>` for download links) and ensure the shared class handles both.
- **Canonical logo is pure CSS in `base.css`**: The NestCheck logo mark (two skewed rectangles) is rendered with `nav .logo-icon span` elements — not an SVG or image. The logo lives exclusively in the `<nav>` in `_base.html`; do not duplicate it in report headers or other components. The icon uses `var(--color-brand)` (#0B1D3A) with the second rectangle at `opacity: 0.6`. Since the nav has a white background, the dark rectangles need no color override. If the nav background ever changes back to dark, the logo icon colors must be updated to maintain contrast.
- **Sticky sidebar scroll containment**: Any `position: sticky` sidebar/rail must have `max-height: calc(100dvh - <top>px)` + `overflow-y: auto` + `overflow-x: hidden` + `min-width: 0`. Without max-height, tall content extends beyond the viewport with no independent scroll. Without `min-width: 0`, CSS grid children use `min-width: auto` which prevents shrinking below content size, causing page-level horizontal scroll. On mobile breakpoints where the sidebar becomes `position: static`, reset with `max-height: none; overflow-y: visible`. Reference: `.report-sidebar` in `report.css`, `.report-layout__rail` in `snapshot.css`.

## Key Patterns

### Async Evaluation (job queue)
- **POST /** with address: creates a job in SQLite, returns `{job_id}` immediately (no client timeout).
- **GET /job/<job_id>**: returns `{status, current_stage?, snapshot_id?, error?}` for polling. status: `queued` | `running` | `done` | `failed`.
- **worker.py**: one background thread per gunicorn worker (started via `gunicorn_config.py` post_fork). Polls for queued jobs, claims atomically, runs `evaluate_property(listing, api_key, on_stage=...)`, updates `current_stage` in DB, then saves snapshot and marks job done or failed.
- Frontend: form submit → POST → get job_id → poll GET /job/{id} every 2s → show stage text → on done redirect to /s/{snapshot_id}.

### API Clients
- GoogleMapsClient wraps all Maps API calls
- Always use `timeout=API_REQUEST_TIMEOUT`
- Handle quota errors gracefully
- **Non-physical place filtering** (NES-211): `places_nearby()` and `text_search()` centrally filter out permanently closed and online-only businesses before caching/returning results. All downstream consumers get clean data without their own filter logic. To add a new online-only service, add to `_ONLINE_BUSINESS_EXACT_NAMES` (preferred) or `_ONLINE_BUSINESS_NAME_PATTERNS` (for regex). Filtered places are attached to `TraceContext.filtered_places` for audit.
- **Auditable heuristic filters**: When filtering API results by name/pattern heuristics, return a reason string (not just bool) so filtered items can be traced and false positives audited. Pattern: `_non_physical_reason() → Optional[str]` with typed reasons like `"closed_permanently"`, `"online_exact:<name>"`, `"online_pattern:<regex>"`.
- **Google Maps `radius` parameter is a location bias, not a strict filter** (NES-250): `text_search()` and `places_nearby()` with a `radius` parameter bias results toward the location but do NOT guarantee results stay within the radius. A query for "city center" near Ann Arbor can return "Southfield City Centre" 35 miles away if Google ranks it highly. Always validate result proximity with `_distance_feet()` (haversine, zero API cost) before accepting. When searching for generic concepts ("downtown", "city center"), prefix the city name via `reverse_geocode_locality()` to improve relevance.
- **Supplemental Text Search for chain coverage** (NES-258): Google Places Nearby Search returns max 20 results ranked by "prominence," which can exclude well-known chains (Kroger, Meijer) in suburban areas. Add a `text_search("supermarket", lat, lng, radius_meters=8000)` call alongside the type-based `places_nearby` calls — Text Search uses relevance ranking, surfacing different results. Pattern: wrap in `try/except` with `logger.warning` (supplemental, never blocks scoring), then merge into the same `all_stores` list before `_dedupe_by_place_id()`. The same pattern applies to any dimension where the 20-result Nearby Search cap drops known venues. Both `text_search()` and `places_nearby()` have per-evaluation caches — initialize both in `GoogleMapsClient.__init__`, not lazily via `hasattr`.
- **Reuse `requests.Session` for connection pooling**: `OverpassHTTPClient` uses a persistent `self._session` (created once in `__init__`) instead of a fresh `requests.Session()` per request. This enables TCP/TLS connection reuse across queries within the same gunicorn worker process. `requests.Session` is thread-safe for concurrent usage. Set `trust_env = False` on the session to skip proxy env var lookups.
- **OSM amenity elements can be nodes OR ways**: Amenities like playgrounds and dog parks are frequently mapped as area polygons (`way`) in OSM, not just point nodes (`node`). When querying Overpass for amenity features, include both `node` and `way` element types. Parse amenity tags from all element types — restricting to `node` only will miss area-mapped features, which are common in suburban parks.

### Venue Scoring Calibration (scoring_config.py + property_evaluator.py)
- **Venue eligibility thresholds**: `VENUE_MIN_REVIEWS` and `VENUE_MIN_RATING` dicts in `scoring_config.py` gate which venues enter scoring. Venues below thresholds are still returned for display — only excluded from headline selection and dimension score computation.
- **Search types ↔ eligibility types contract**: Every Google Places type accepted in the eligibility filter (e.g., `["cafe", "coffee_shop", "bakery"]` at line 4280) must have a corresponding `places_nearby()` query upstream. A type in the filter but missing from the search silently produces zero matches for that category — no error, just empty results. When adding a new accepted type, add the API query in both `score_third_place_access()` and `get_neighborhood_snapshot()`.
- **Quality ceiling pattern** (Phase 2): Walk-time proximity scores are capped by a diversity ceiling computed from two signals: (1) distinct social-category buckets via `_SOCIAL_BUCKET_MAP` → `THIRD_PLACE_CATEGORY_CEILINGS`, and (2) median review depth ±1 adjustment. `final_score = min(walk_time_score, ceiling)`. The ceiling is structurally bounded by search query types — only categories returned by the Google Places search can contribute buckets.
- **Suppressed dimensions** (`points=None`): When no venues pass eligibility, return `Tier2Score(points=None, suppressed_reason=...)` instead of `points=0`. Template shows "—", composite scoring excludes the dimension from aggregation (reduces effective max). This avoids penalizing addresses for data gaps.

### Narrative Insights (NES-191+)
- Insight generators are pure functions: `dict → str | None`. Keep them side-effect-free for testability.
- Place insight logic in dedicated modules (not `app.py`) — `app.py` is for routes/views.
- **Dict field-name contracts**: Insight functions that consume dicts from `present_checks()` must use the field names that function actually produces (`"name"`, `"result_type"`, `"category"`, etc.) — not alternative names like `"display_name"` or `"check_id"`. When adding a new consumer of presented-check dicts, verify field names against `present_checks()` output, not against `_PROXIMITY_LABELS` keys or other lookup tables. Always add a `or "this hazard"` (or similar) terminal fallback when interpolating dict values into user-facing sentences to prevent sentence fragments from malformed input.
- Avoid duplicating utility helpers across modules (e.g., Oxford-comma join) — use a shared module.
- **`_PROXIMITY_LABELS` must use "a/an + singular noun" pattern**: Every entry is interpolated into sentences like `"This address is close to {label}"` and joined with `_join_and()`. Plural or article-free entries (e.g., `"high-voltage power lines"` instead of `"a high-voltage power line"`) produce awkward grammar when combined with other labels that do have articles.
- **Test assertions on prose should be semantic, not literal**: Insight generator tests that assert on exact prose fragments (`"driving"`, `"on the other hand"`, `"no parks"`) break during content design passes. Prefer assertions on structural properties (entity names present, `car_dependent` flag, key semantic words like `"car"`) over verbatim copy. When a content pass updates N prose strings, expect ~N test assertion updates — budget for it.
- **HTML-escape external data in `| safe` narrative generators**: Functions that return HTML strings rendered with Jinja's `| safe` filter (e.g., `generate_report_narrative`, any future insight generator returning HTML) must escape all external data (place names from Google Places API, user input, check headlines) via `markupsafe.escape()` before interpolation. Without this, a malicious place name like `<script>alert(1)</script>` executes in the browser. Pattern: `str(_html_escape(name))` at the point of extraction, not at interpolation.
- **`_join_labels()` requires a non-empty list**: The function has no internal guard for empty lists — `_join_labels([])` crashes with `IndexError`. When the input list is built from a filter that can reduce to zero elements (e.g., `[i for i in items if i]`), always guard the call: `if filtered: _join_labels(filtered)`. This is easy to miss when the upstream list is truthy but contains only falsy elements like empty strings.

### Check Display Metadata (app.py)
- Each Tier1Check `name` in `property_evaluator.py` needs entries in app.py: `_SAFETY_CHECK_NAMES`, `_CLEAR_HEADLINES`, `_ISSUE_HEADLINES`, `_WARNING_HEADLINES`, `_HEALTH_CONTEXT`.
- Only add headline entries for result states the check can actually produce. `_build_health_context()` returns `None` silently for missing keys.
- **Naming split**: Legacy checks use display names (`"Power lines"`, `"Gas station"`). Phase 1B spatial checks use `snake_case` (`"ust_proximity"`, `"hifld_power_lines"`). Match the convention of the source check function.
- New checks NOT in `_CHECK_SOURCE_GROUP` render individually (not collapsed).
- **Hazard tier classification** (NES-241): `_TIER_2_CHECKS` in `app.py` defines area-level indicators (EJScreen) that render in a compact collapsible section. All other safety checks default to Tier 1 (prominent cards). When adding a new safety check, it's Tier 1 by default — only add to `_TIER_2_CHECKS` if it's an area-level statistical indicator rather than a direct proximity hazard. The `hazard_tier` field is computed in `present_checks()` and backfilled in `view_snapshot()` for old snapshots. Template filters on `selectattr('hazard_tier', 'equalto', N)` — missing the field silently excludes the item from both tiers, so the backfill is critical.
- **Tier1Check serialization paths**: `Tier1Check` is serialized in three places — `result_to_dict()` (main snapshot), the compare route's serialization loop, and CSV export. When adding a new field to the dataclass, update ALL three. The compare path is easy to miss.
- **Nullable field propagation**: When widening a dataclass field from non-optional to `Optional` (e.g., `points: int → Optional[int]`), audit every consumer: arithmetic aggregation (`sum()`), string formatting (`f"{x} pts"`), template division (`score / max`), and all serialization paths. Each needs a null guard or filter. The `_scorable = [s for s in scores if s.points is not None]` pattern keeps aggregation clean.
- **Search radius vs threshold** (NES-203): Search radius can be wider than the warning threshold to report "Nearest: X (Y ft)" on PASS. Set `show_detail=True` on those PASS results. When nothing is found even within the expanded radius, the detail string should match the `_CLEAR_HEADLINES` wording (use the threshold distance, not the search radius).
- **Cascading distance filters must use explicit upper bounds** (NES-266): When a proximity check has tiered zones (FAIL ≤ 150m, WARNING 150–300m, PASS > 300m), every candidate filter must have BOTH a lower AND upper distance bound. A filter like `distance > FAIL_RADIUS and aadt >= THRESHOLD` without `and distance <= WARN_RADIUS` silently extends the warning zone to the full search radius (e.g., 600m instead of 300m). The unused threshold constant makes it look correct at a glance. Ground-truth validation caught this bug — unit tests didn't because the test distance (200m) was inside the intended range.
- **Presentation-layer suppression** (NES-196): To hide checks from display without changing evaluation or storage, filter in the route handler (e.g., `view_snapshot()`), not in `present_checks()` or `result_to_dict()`. Use `result = {**result, ...}` to shallow-copy the deserialized snapshot dict before modifying — never mutate it in-place, as a future caching layer would silently corrupt shared state. Suppressed metadata (e.g., count) should travel inside `result` dict, not as a separate template var, so Jinja includes and macros can access it without relying on inherited scope. Apply suppression to ALL routes that render the same data (snapshot, compare-health, etc.) to avoid inconsistent UX.
- **Confidence score caps** (NES-sparse-data): Dimension scores from data-confidence-aware scorers must be passed through `_apply_confidence_cap(score, confidence)` before building the `Tier2Score`. `estimated` caps at 8/10, `verified` uncapped, `not_scored` excluded from composite entirely. Apply consistently to every dimension that classifies confidence — omitting it lets sparse data produce artificially high ratings.
- **`apply_piecewise` returns float → round to int for `Tier2Score.points`**: `apply_piecewise()` and `apply_quality_multiplier()` return floats, but `Tier2Score.points` is `Optional[int]`. Every scoring path must round at the `Tier2Score` boundary: `points = int(max(cfg.floor, capped_score) + 0.5)`. The `score_road_noise` function established this pattern; all other dimensions using `apply_piecewise` must follow it. Omitting the round silently stores floats in an int field, causing "7.6/10" display artifacts and type-contract violations in serialization paths.
- **Snapshot migration functions must run on shallow copies**: Every route that deserializes a snapshot and runs migration functions (`_migrate_dimension_names`, `_migrate_confidence_tiers`) must shallow-copy first: `result = {**snapshot["result"]}`. There are currently 4 deserialization paths: `view_snapshot()`, `export_snapshot_json()`, `export_snapshot_csv()`, and the compare route. When adding a new migration function, wire it into ALL 4 paths — missing one causes silent data inconsistencies. The `view_snapshot` route copies via `{**result, ...}` during suppression; the others need explicit `{**snapshot["result"]}`.
- **Shallow copy only covers top-level keys**: `{**result}` copies the top-level dict, but nested structures (lists of dicts like `dimension_summaries`, nested dicts like `green_escape`) remain shared references to the stored snapshot. When backfilling keys on nested dicts, either shallow-copy the nested structure first (`dict(_ge)`, `[{**dim} for dim in dims]`) and reassign, or use a non-mutating pattern like `{**dim, "new_key": value}` in a comprehension. Mutating nested dicts in-place silently corrupts the stored snapshot under a future caching layer.
- **Shared helpers for serialization + backfill**: When `result_to_dict()` and `view_snapshot()` backfill need identical classification logic (e.g., `_dim_band()`), extract to a module-level function. Defining the helper inside `result_to_dict()` forces the backfill to duplicate the logic inline, which diverges silently when thresholds change. Pattern: module-level `_DIM_BANDS` tuple + `_dim_band()` function, called from both the serialization list comprehension and the backfill loop.
- **Backfill ordering and safe template defaults**: Functions that compute derived fields from migrated data (e.g., `_compute_show_numeric_score` consuming `data_confidence`) must run *after* the relevant migration (`_migrate_confidence_tiers`). Do not add legacy tier names to the derived function's allowlist as a substitute for correct ordering — that masks wiring bugs silently. Template fallbacks for backfilled display flags should use the *restrictive* value (`false` / hide), so a missing backfill surfaces visibly rather than silently showing incorrect data.
- **Composite scoring exclusion for not_scored**: Dimensions with `data_confidence == CONFIDENCE_NOT_SCORED` are excluded from both the numerator and denominator of composite scoring. This prevents both inflation (invented scores pulling average up) and deflation (zero scores pulling average down). The `_scorable` filter in `evaluate_property()` handles this.
- **Dimension name renaming** (NES-210): When renaming a `Tier2Score.name`, update ALL construction sites in the scoring function, update `TIER2_NAME_TO_DIMENSION` (keep old name as backward-compat alias), and add the old→new mapping to `_LEGACY_DIMENSION_NAMES` + `_migrate_dimension_names()` in app.py. The migration must run in ALL snapshot deserialization paths: `view_snapshot()`, `export_snapshot_json()`, `export_snapshot_csv()`, and the compare route. In `view_snapshot()`, run migration AFTER the `{**result, ...}` shallow copy to avoid mutating stored snapshot dicts.
- **Pre-computed sidebar variables**: `_result_sections.html` pre-computes `show_score`, `band_class`, `band_label`, `safety_checks`, `clear_count`, `warning_count`, `issue_count`, `safety_concerns` at template top for reuse in both main column and sidebar. When adding new sidebar data, compute it in the top block — don't duplicate Jinja filter chains inside both column and sidebar sections.
- **Composite scoring is equal-weight**: All scorable dimensions contribute equally to the 0-100 composite score (`tier2_total / tier2_max * 100`). The persona/weighted scoring infrastructure (NES-133) was removed. `TIER2_NAME_TO_DIMENSION` is retained solely for the compare route's dimension ordering.

### SQLite Concurrency (models.py)
- `_get_db()` sets `PRAGMA busy_timeout=30000` (30s) for write contention under concurrent gunicorn workers + evaluation threads.
- Job-critical write functions (`create_job`) retry up to 3× on `OperationalError` with "locked"/"busy" in the message, with progressive backoff. Read functions (`get_job`) retry once.
- Pattern for retry-safe DB functions: open connection inside `try`, use `finally: conn.close()`, catch `sqlite3.OperationalError` outside the connection block.
- `sqlite3.connect(timeout=N)` and `PRAGMA busy_timeout` both set the busy timeout — use only the PRAGMA (more explicit, avoids redundancy).

### Data Caches (models.py)
- Pattern: `get_<name>_cache(key) → Optional[str]` / `set_<name>_cache(key, json) → None`
- Always swallow cache errors so they never break an evaluation.
- Each cache table has its own TTL constant (`_<NAME>_CACHE_TTL_DAYS`).
- Existing caches: weather (30-day), census (90-day).

### Ground-Truth Calibration (scripts/generate_ground_truth_*.py + scripts/validate_ground_truth_*.py)
- **Generator/validator pairs**: Each spatial check has a generator (creates synthetic test points from real spatial.db data) and a validator (runs the actual check function and compares results). UST: point geometry, single threshold axis (distance). HPMS: linestring geometry, dual threshold axes (distance AND AADT).
- **Nearest-neighbor adjustment**: After generating a test point at a controlled distance from one facility, query the spatial index for the ACTUAL nearest facility. If a different facility is closer, adjust the expected result. Without this, dense areas produce false mismatches.
- **Threshold import pattern**: Try importing constants from `property_evaluator.py`; fall back to hardcoded values if the import fails (Flask side-effects). Print which path was taken. The hardcoded values MUST stay in sync — add a comment with the canonical source file and line.
- **LINESTRING perpendicular offset**: For linestring data (HPMS), generate test points perpendicular to the segment bearing at the centroid. Use `ST_StartPoint`/`ST_EndPoint` for bearing, `ST_Centroid` for midpoint, then offset at bearing + 90°. The actual `ST_Distance` to the line may differ slightly from the offset distance because the test point's nearest approach may not be the centroid.
- **Dual-axis test design** (HPMS): Partition segments into pools by the non-distance axis (AADT above/below threshold). HIGH_TRAFFIC pool gets 3 distance variants (CLOSE/MIDDLE/FAR). LOW_TRAFFIC pool gets 1 variant (CLOSE only) to test that the non-distance condition prevents triggering.
- **Aggregate validation entrypoint**: `scripts/validate_all_ground_truth.py` discovers all `validate_ground_truth_*.py` scripts, matches each to `data/ground_truth/<dimension>.json`, runs them as subprocesses, and prints an aggregate summary. Adding a new dimension requires only a generator script, validator script, and ground-truth file — no changes to the aggregate runner. The runner parses `Matches:` / `Mismatches:` lines from validator stdout, so individual validators must preserve that output format.
- **Ground truth file location**: Committed ground-truth files live in `data/ground_truth/<dimension>.json` (not at the project root). Generated with fixed seeds (`--seed 42`) for determinism. Regenerate after changing thresholds, expanding spatial data, or widening geographic coverage.

### Spatial Ingest (startup_ingest.py + scripts/ingest_*.py)
- `startup_ingest.py` calls `ingest()` directly with keyword args — scripts must accept kwargs, not just argparse.
- State filter format varies by dataset: 2-letter (`"NY"`) for EJScreen/TRI/NCES, full name (`"New York"`) for UST, FIPS code (`"36"`) for TIGER, postal abbreviation (`STABR='NY'`) for NCES. Always check the upstream API field.
- ArcGIS bbox filtering: `geometry` (JSON envelope), `geometryType=esriGeometryEnvelope`, `inSR=4326`. Server-side filter.
- Wiring pattern: lazy-import wrapper → `_table_has_data()` → `_run_ingest()`. One failure never blocks others.
- **ArcGIS polygon ring orientation**: Clockwise = outer boundary, counter-clockwise = hole. Use shoelace formula (`_ring_is_clockwise()`) to classify rings when features may have disjoint parts (e.g., school districts with enclaves). The simpler `MULTIPOLYGON(((ring1),(ring2)))` pattern used by SEMS/FEMA only works when all rings belong to one contiguous polygon.
- **Two-table join pattern** (NES-206): Spatial polygon table (`facilities_school_districts`) for point-in-polygon → extract GEOID → join to a separate lookup table (`state_education_performance`) for enrichment data. Table contains multi-state data (NY, CT, NJ) with a `state` column; GEOID is unique across states. Different from single-table spatial checks.
- **Bundled CSV for data without stable APIs**: When upstream data is only available as Access DBs or manual downloads (e.g., NYSED), ship a curated CSV in `data/` and flag as `MANUAL REFRESH` in the `dataset_registry` notes. Include refresh cadence.
- **Point geometry ingest**: For point-location datasets (e.g., NCES schools), use `MakePoint(lon, lat, 4326)` instead of `GeomFromText('POINT(...)', 4326)`. Simpler and avoids WKT string building. Geometry comes as `{x: lon, y: lat}` from ArcGIS; always fall back to attribute fields (`LATCOD`/`LONCOD` for NCES 2022-23) if geometry is missing.
- **ArcGIS field name drift across dataset years**: Federal ArcGIS REST endpoints (NCES EDGE, TIGERweb) rename fields between releases without versioning. NCES 2022-23 changed `LEVEL` → `SCHOOL_LEVEL`, `LAT`/`LON` → `LATCOD`/`LONCOD`, and `SCH_TYPE_TEXT` became unqueryable in `outFields` (returns 400). When an ingest script fails with ArcGIS `{'code': 400, 'message': 'Failed to execute query.'}`, check field names with `outFields=*&resultRecordCount=1` first — don't assume the schema from a prior year still holds. Run `--discover` mode before full ingestion after any dataset year bump.
- **NCES suppression codes in numeric fields**: NCES uses negative integers (`-1`, `-2`) as suppression/missing-data indicators in otherwise-numeric fields like `TOTFRL` (Free/Reduced Lunch count). Always filter `< 0` values before arithmetic — they silently produce nonsensical results (e.g., `-0.3%` FRL rate) rather than raising errors.
- **Derived percentages from upstream counts**: When computing percentages from two upstream fields (e.g., TOTFRL/MEMBER for FRL%), cap at 100% (data quality issues can produce numerator > denominator) and filter out negative suppression codes (see above).
- **Template component duplication**: When the same UI component renders in multiple conditional branches (e.g., school card inside vs. outside school district section), extract into a Jinja macro in `_macros.html` immediately. Copy-paste diverges silently on the next edit.
- **Jinja macros with inline HTML strings need `| safe`**: When a macro builds an HTML string (e.g., an SVG icon) via `{% set _icon_svg = '<svg ...>' %}`, Flask's autoescape (enabled for `.html` templates) will escape `{{ _icon_svg }}` into literal `&lt;svg&gt;` text. Always use `{{ _icon_svg | safe }}` for internally-constructed HTML constants. This is safe because the string is hardcoded in the macro, not derived from user input. Never apply `| safe` to variables that contain external data.
- **Multi-state shared table idempotency**: When multiple scripts write to the same table partitioned by `state` column, never use `DROP TABLE` — use `CREATE TABLE IF NOT EXISTS` + `DELETE FROM ... WHERE state = 'X'` + INSERT. The DROP pattern destroys other states' data when run standalone.
- **Per-state data checking in startup_ingest**: When a table has rows from multiple independent ingestion sources (e.g., `state_education_performance` with NY/NJ/CT), use `_table_has_state_data(db_path, table, state)` to check each state independently. Using `_table_has_data()` (checks total rows) will skip ingestion for state B if state A's data already exists.
- **Catch `IntegrityError` in CSV ingest loops**: When inserting from bundled CSVs into tables with UNIQUE constraints, always catch `sqlite3.IntegrityError` alongside `ValueError`/`KeyError`. A duplicate key in the CSV would otherwise crash the entire ingest, leaving partial data.
- **HPMS display-name resolution** (`_build_road_display_name`): HPMS `ROUTE_ID` is a coded segment identifier (e.g., "00000001__", "14000638__"), not a road name. Use `_looks_like_route_id()` to detect coded IDs — checks for `"__"` (double underscore, universal in HPMS route_ids across states) and pure-digit strings. Fallback chain: `route_name` (e.g., "SUNRISE HWY", "US 1") → `segment.name` (if not a route_id) → `"Route {route_number}"` → `route_id` as last resort. When building Tier1Check details for any result path (FAIL, WARNING, PASS-with-detail), always call `_build_road_display_name()` — don't inline the name resolution. Multiple code paths constructing the same detail string will diverge when the resolution logic changes. **NJ caveat**: NJ HPMS route_ids use `"00000001__"` format (8 zero-padded digits + `__`), which fails Python's `str.isdigit()`. Always use `_looks_like_route_id()` instead of `isdigit()` for route_id detection.
- **HPMS "Ref Rt" prefix cleaning**: NY HPMS encodes reference routes as `"Ref Rt 907M GRAND CNTRL PKWY"`. The ingest script strips the `"Ref Rt "` prefix and the numeric code (e.g., `"907M "`) to extract the human name. Other states may use different conventions.
- **Bundled CSV GEOIDs must match TIGER exactly**: The two-table join (`facilities_school_districts` → `state_education_performance`) uses string equality on GEOID. Fabricated or misformatted GEOIDs silently produce zero matches with no error. When adding or updating bundled education CSVs, verify GEOIDs against the TIGER WMS endpoint (`MapServer/14/query?where=GEOID='NNNNNNN'`). Use `scripts/crosswalk_geoids.py` as the reference crosswalk tool.

### Comparison View (app.py + compare.html)
- **Structured differential pattern** (NES-207): Multi-snapshot comparison views compute differential data in a pure helper (`_build_comparison_data()`) called from the route handler, not in Jinja templates. The helper returns typed data structures (health grid, dimension rows, key differences) — the template only renders. This keeps business logic testable and templates simple.
- **Legacy/Phase 1B dedup in comparison grids**: When the same check exists in both legacy (`"Power lines"`) and Phase 1B spatial (`"hifld_power_lines"`) forms, use `_SPATIAL_SUPERSEDES` to skip legacy rows when the spatial version exists in any snapshot. Also deduplicate by display label (`seen_labels`) to prevent duplicate rows.
- **`score_ring` macro requires `report.css`**: The `score_ring` macro from `_macros.html` depends on `.band-exceptional`, `.band-strong`, etc. classes defined in `report.css`. Any template using the macro must load `report.css` even if it doesn't render full report sections.
- **Comparative verdict** (NES-218): `_build_comparative_verdict()` is a pure function that synthesizes headline + body from already-computed comparison data. It does NOT call APIs or touch the DB. When adding new comparison prose, add a branch to this function and a corresponding unit test — don't put text-generation logic in the template.
- **Branch-priority ordering for score spreads**: When multiple `if spread <= N` branches exist, check tighter thresholds first (e.g., `<= 3` before `<= 5`) to avoid the looser condition swallowing the tighter one. This is a common bug in cascading numeric thresholds — always order from most specific to least specific.

### Quality Ceiling (scoring_config.py + property_evaluator.py)
- **Quality ceiling pattern**: `QualityCeilingConfig` caps a dimension's max score based on venue diversity + review depth. Applied *before* `_apply_confidence_cap()` so both constraints compose. To add a ceiling to another dimension: add `quality_ceiling=QualityCeilingConfig(...)` to its `DimensionConfig`, then call `_compute_quality_ceiling()` between the raw score and the confidence cap.
- **Threshold reachability**: Config thresholds must be reachable by the upstream classifier. If `_classify_coffee_sub_type()` returns at most 3 distinct types, a `(4, bonus)` diversity threshold is dead code. Always verify the input domain before defining threshold tiers.
- **Pre-filtered inputs shift threshold floors**: `eligible_places` is already filtered to `reviews >= 30` and `rating >= 4.0` before reaching the ceiling. Depth thresholds near the filter floor (e.g., `(50, 1.0)`) will almost always trigger. Account for upstream filters when tuning thresholds.

### Display Thresholds (scoring_config.py)
- `WALK_DRIVE_BOTH_THRESHOLD` (20), `WALK_DRIVE_LEAD_DRIVE_THRESHOLD` (25), and `WALK_DRIVE_ONLY_THRESHOLD` (40) in `scoring_config.py` are the canonical walk/drive time display thresholds. Four display tiers: walk-only (≤20), walk-first-then-drive (20–25), drive-first-then-walk (25–40), drive-only (>40). All display logic (templates, drive-time fetching) should reference these, not hardcode magic numbers.
- **Scoring thresholds ≠ display thresholds**: `WALK_TIME_MARGINAL` (30 min, `green_space.py`) controls the walk-time *scoring curve*. `WALK_DRIVE_BOTH_THRESHOLD` (20 min) controls when drive times are *fetched and shown*. Don't conflate them — a constant can share a numeric value with a display threshold but serve a different purpose (scoring vs presentation).
- Transit drive-time fetch uses its own hardcoded threshold (`> 20` in `find_primary_transit`) — intentionally independent of the display constants. It follows a walk-primary/drive-secondary pattern where both are always shown when drive time exists.
- **Display-time derivation for cross-cutting widgets** (NES-249): When a UI component synthesises data from multiple dimensions (e.g., walkability summary from coffee/grocery/fitness/parks/transit walk times), derive it at display time in `view_snapshot()` rather than storing it in the snapshot. Pattern: pure function `_build_*(result) → dict | None`, called once before `render_template()`, assigned to `result["widget_key"]`. Benefits: works for old snapshots without migration, zero API cost, no `result_to_dict()` change, no export/compare wiring. Only add to `view_snapshot()` — not exports or compare routes.
- **`_best_walk_time()` filters by `rating is not None`**: Placeholder entries in `neighborhood_places` (e.g., "No full-service grocery nearby") have `rating: None` and `walk_time_min: 0`. Filter on `rating is not None` to exclude them. This doesn't apply to `green_escape.best_daily_park` or `urban_access.primary_transit` which are already validated by the evaluator.

### Payment (Stripe Checkout + Free Tier)
- **Payment state machine**: `pending` (checkout created) → `paid` (webhook confirmed or Stripe API verified) → `redeemed` (evaluation started). On eval failure: `redeemed` → `failed_reissued` (user can retry). Atomic CAS guards (`expected_status`) prevent TOCTOU races between webhook and return-from-Stripe.
- **Redeem-then-link gap**: `redeem_payment(token)` is called before `create_job()` (to gate creation), then `update_payment_job_id(token, job_id)` links them after. Server crash between these calls leaves a redeemed payment with no job_id — recoverable manually but not automatically.
- **Stripe return URL must URL-encode query params**: `success_url` carries `address`, `email`, `place_id` back through Stripe's redirect. These are user-supplied strings — use `urllib.parse.quote(value, safe='')` to prevent `&`, `=`, `#`, spaces from breaking URL parsing on return.
- **Cached snapshots bypass free tier recording**: The snapshot cache check (`is_snapshot_fresh`) can short-circuit before `record_free_tier_usage()`. This means repeated requests for a cached address don't consume the free credit — by design, since no new evaluation runs.
- **CSRFProtect instance must be stored**: `CSRFProtect(app)` returns the instance needed for `@csrf.exempt()`. Store in `_csrf` module variable. Webhook endpoint uses Stripe signature verification instead of CSRF.
- **Stripe graceful degradation**: Same pattern as OAuth — conditional import + module-level flag (`STRIPE_AVAILABLE`). When `stripe` package isn't installed or env vars are missing, checkout route returns 400 and app works in free-only mode.
- **`hash_email()` compute-once pattern**: When the POST `/` route uses email_hash for free tier check, job creation, and free tier recording, compute `email_h = hash_email(email) if email else None` once at the top of the payment gate block and reuse everywhere.

### Authentication (Flask-Login + Google OAuth)
- **`_FlaskUser` is not a DB model**: It wraps the dict from `get_user_by_id()` and implements Flask-Login's interface (`get_id()`, `is_authenticated`, etc.). Created fresh on every request via `user_loader`.
- **Open redirect prevention on OAuth callback**: `next_url` from session must be validated: only allow paths starting with `/` (not `//` which browsers treat as protocol-relative URLs). Pattern: `if not next_url or not next_url.startswith("/") or next_url.startswith("//"): next_url = "/my-reports"`
- **Snapshot claiming runs on every login**: `claim_snapshots_for_user()` matches by email on `WHERE user_id IS NULL AND email = ?`. Must run every login, not just `created=True`, because users may create evaluations with their email while logged out between sessions.
- **All snapshot save paths must include `user_id`**: Three INSERT paths exist: `save_snapshot()`, `save_snapshot_for_place()` (INSERT branch), and `save_snapshot_for_place()` (UPDATE branch — doesn't need `user_id` since it preserves existing). When adding a new column to snapshots, update all INSERT statements. Same pattern as documented for `Tier1Check` serialization paths.
- **Graceful degradation pattern**: Conditional import + module-level flag (`_oauth_enabled`). When OAuth env vars are missing: no Authlib import, no OAuth routes functional (redirect with flash), nav hides Sign In link via `{% if oauth_enabled %}`. App behavior is unchanged.

### Stripe Integration (Payments)
- **Payment status constants**: Use `PAYMENT_PENDING`, `PAYMENT_PAID`, `PAYMENT_REDEEMED`, `PAYMENT_FAILED_REISSUED` from `models.py` — never bare strings. Typos in status strings silently break state transitions with no error.
- **`_get_or_create_stripe_customer()` graceful degradation**: Customer creation failure falls back to `customer_email` pre-fill in the Stripe Checkout session, never blocks checkout. Pattern: try `Customer.create` → on failure, log warning and return `None` → caller checks and uses `customer_email` fallback.
- **Stripe webhook CSRF exemption**: Store `_csrf = CSRFProtect(app)` at module level and call `_csrf.exempt(view_func)` for webhook routes that use their own signature verification (Stripe-Signature header). Without storing the `CSRFProtect` instance, you can't exempt routes.
- **`REQUIRE_PAYMENT=True` requires `STRIPE_AVAILABLE=True`**: Pending payments cannot be verified without Stripe API access. The payment validation block must reject `pending` status when `STRIPE_AVAILABLE` is False — otherwise unverified payments proceed to job creation.
- **Stripe exception import path**: `stripe.error.SignatureVerificationError` was moved to `stripe.SignatureVerificationError` in stripe-python v6+. Use `except Exception as exc: if "SignatureVerification" in type(exc).__name__` for cross-version compatibility.
- **Free tier TOCTOU prevention**: `free_tier_usage.email_hash` must have a UNIQUE constraint. Use `INSERT OR IGNORE` + check `cursor.rowcount > 0` instead of SELECT-then-INSERT, which allows duplicate grants under concurrent requests.

### Safari Mobile / Viewport (iOS)
- `_base.html` sets `viewport-fit=cover` — required for `env(safe-area-inset-*)` to work. Do not remove.
- Never use bare `100vh` for layout heights; always add a `100dvh` override on the next line. Safari iOS `100vh` includes the URL bar's hidden space, which breaks scroll detection and makes the address bar stick in its collapsed state.
- Fixed-position elements pinned to `bottom: 0` must include `padding-bottom: calc(<normal> + env(safe-area-inset-bottom, 0px))` so they clear Safari's bottom bar and home indicator. Currently applied to: cookie banner (`base.css`), compare tray (`_compare_tray.html`).
- When adding new `position: fixed; bottom: 0` elements, follow the same `env()` pattern.

## Workflow Orchestration

### 1. Plan Mode Default
- Enter plan mode for ANY non-trivial task (3+ steps or architectural decisions)
- If something goes sideways, STOP and re-plan immediately — don't keep pushing
- Use plan mode for verification steps, not just building
- Write detailed specs upfront to reduce ambiguity

### 2. Subagent Strategy
- Use subagents liberally to keep main context window clean
- Offload research, exploration, and parallel analysis to subagents
- For complex problems, throw more compute at it via subagents
- One task per subagent for focused execution

### 3. Self-Improvement Loop
- After ANY correction from the user: update CLAUDE.md with the pattern
- Write rules that prevent the same mistake
- Ruthlessly iterate on these lessons until mistake rate drops

### 4. Verification Before Done
- Never mark a task complete without proving it works
- Diff behavior between main and your changes when relevant
- Ask yourself: "Would a staff engineer approve this?"
- Run tests, check logs, demonstrate correctness

### 5. Demand Elegance (Balanced)
- For non-trivial changes: pause and ask "is there a more elegant way?"
- If a fix feels hacky: "Knowing everything I know now, implement the elegant solution"
- Skip this for simple, obvious fixes — don't over-engineer

## Core Principles

- **Simplicity First**: Make every change as simple as possible. Impact minimal code.
- **No Laziness**: Find root causes. No temporary fixes. Senior developer standards.
- **Minimal Impact**: Changes should only touch what's necessary. Avoid introducing bugs.

## When Unsure

- Ask clarifying questions before implementing
- Check existing patterns in the codebase first
- Prefer simple solutions over clever ones
